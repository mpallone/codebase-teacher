You are evaluating the quality of documentation generated by `codebase-teacher` for a real-world open-source repository. Below you will find:

1. The repo's README and source code samples (ground truth)
2. The generated documentation and diagrams (what you are judging)

## Scoring Dimensions (1-5 each)

- **factual_accuracy**: Are the claims in the generated docs actually true? Cross-reference against the source code. A score of 1 means major hallucinations; 5 means every claim is verifiable.
- **completeness**: Are the major components, modules, APIs, and data flows covered? 1 means most things are missing; 5 means comprehensive coverage.
- **architecture_clarity**: Does the architecture doc give a clear mental model of how the codebase is structured? 1 means confusing or wrong; 5 means a new engineer would understand the system after reading it.
- **api_coverage**: Are HTTP endpoints, routes, handlers, and their methods/paths correctly identified? 1 means most routes are wrong or missing; 5 means thorough and accurate. Score N/A if the repo has no APIs.
- **infrastructure_coverage**: Are databases, queues, caches, cloud services, and external dependencies identified? 1 means major infrastructure is missed; 5 means complete. Score N/A if the repo has no infrastructure.
- **diagram_quality**: Are Mermaid diagrams syntactically valid and useful? Do they accurately represent the system? 1 means broken or misleading; 5 means clear and accurate.

## Verdict

- **pass**: All applicable scores >= 4. The documentation is good enough to onboard an engineer.
- **needs_work**: Any applicable score is 2-3. Usable but has meaningful gaps or inaccuracies.
- **fail**: Any applicable score is 1. Fundamental problems that make the docs misleading.

## Required Output Format

Provide your evaluation as:

1. **Scores**: List each dimension with its numeric score and a one-sentence justification.
2. **Verdict**: pass / needs_work / fail
3. **Findings**: Specific issues found, each with:
   - severity: high / medium / low
   - description: What's wrong
   - evidence: Quote or reference from the source code that proves the issue
4. **Hallucinations**: Any claims in the generated docs that are not supported by the source code.
5. **Missing concepts**: Important things the docs should cover but don't.
6. **What worked well**: 1-3 things the generated docs got right (to preserve in future iterations).
