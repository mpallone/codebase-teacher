What is this?

Conduit is a Medium-style social publishing platform — think articles, comments, tags, author profiles, follows, and favorites — exposed as a clean REST API. It is a full implementation of the RealWorld spec, which means the API contract is standardized and any compliant front-end can plug straight in.

The repo ships two things: the production application server (async Python, PostgreSQL-backed) and an automated end-to-end test suite that validates the live API with real HTTP calls. There is no shared library between them — they communicate purely over the network.

Why does it exist?

RealWorld implementations exist so teams can evaluate a realistic, non-trivial codebase — not a "hello world" tutorial, but not a sprawling enterprise system either. Conduit hits the sweet spot: it has auth, relationships, pagination, and business logic, while remaining small enough to read in a day.

Concrete example: Priya is evaluating async Python web frameworks for her team. She clones this repo, runs docker-compose up, and within minutes has a fully working API she can poke at via the Swagger UI at /docs. She then runs ./postman/run-api-tests.sh to verify everything passes, reads the route handlers to see how JWT auth is wired, and leaves with a concrete opinion — all without writing a single line of code herself.

High-level walkthrough

The five major pieces:

Piece What it does
app/api/routes/ Thin HTTP controllers — one file per domain (users, articles, comments, profiles, tags)
app/api/dependencies/ The gating layer — resolves JWT tokens, looks up entities by slug/username, enforces ownership before handlers run
app/services/ Stateless utilities: slug generation, password hashing, JWT encoding, uniqueness checks
app/db/repositories/ All database I/O — queries PostgreSQL and returns typed Python objects
postman/ Shell script + Postman collection that fires real HTTP requests against a live server to prove the API works

How a request flows:

flowchart LR
    Client -->|HTTP + Bearer token| Routes
    Routes -->|Depends| Dependencies
    Dependencies -->|JWT decode + entity lookup| Services
    Dependencies -->|DB lookup| Repositories
    Routes -->|business call| Repositories
    Repositories -->|asyncpg| PostgreSQL[(PostgreSQL)]
    Routes -->|Pydantic serialization| Client

Walk through a concrete case — POST /api/articles:

  1. Routes receive the request and declare their dependencies.
  2. Dependencies decode the JWT, fetch the current user, and reject the request with 401/403 if anything looks wrong — before the handler even runs.
  3. The route handler calls ArticlesRepository.create_article, passing the validated payload.
  4. Repository opens a transaction, inserts the article and its tags, then fires a few sub-queries to build a fully denormalized Article object (tags, favorites count, author profile embedded).
  5. The handler wraps that in a Pydantic response envelope and returns JSON.

The Postman test suite sits entirely outside this path — it just plays the role of a real client, exercising the full stack from the outside.

⚠️ One known rough edge: the article read path has an N+1 query pattern — each article row triggers 3–4 sub-queries to populate its embedded author and tag data. It works fine at low volume; keep it in mind if you're profiling.

1. System Overview

Conduit is a production-ready REST API that implements the RealWorld spec. It provides a social publishing backend with:

  • User accounts — registration, login, profile management
  • Articles — full CRUD with URL slugs, tags, and a pagination-aware list endpoint
  • Comments — threaded on articles
  • Social graph — follow/unfollow users; personalized feed
  • Favorites — bookmark articles

The system ships as two separate deployable units that share only the HTTP contract:

Component What it is
app/ The FastAPI ASGI server — the application itself
postman/ A shell-script + Postman collection that tests the API end-to-end

How credentials are made safe for concurrent runs:

# From run-api-tests.sh (paraphrased)
TIMESTAMP=$(date +%s)
USERNAME="u${TIMESTAMP}"         # e.g. u1713340800
EMAIL="u${TIMESTAMP}@mail.com"

This prevents two parallel CI runs from colliding on the same username/email during user registration.

Operational notes:

  • A 500 ms inter-request delay is injected between Newman requests (--delay-request 500). This suggests either API rate-limiting sensitivity or timing dependencies between chained requests (e.g., a slug created in request N must exist before request N+1 reads it).
  • Test data (users, articles, comments) accumulates permanently in the target database. There is no teardown.
  • The APIURL environment variable controls which environment is targeted. Override it to point at local, staging, or production.

4. Internal Layering of the App Server

The application is structured as four strict vertical layers. A request must pass through each in order:

flowchart TD
    A["HTTP Request"] --> B["CORS Middleware\n(Starlette)"]
    B --> C["DI Layer\napp/api/dependencies/"]
    C --> D["Route Handler\napp/api/routes/"]
    D --> E["Services\napp/services/"]
    D --> F["Repositories\napp/db/repositories/"]
    F --> G["aiosql + PyPika\napp/db/queries/"]
    G --> H[("PostgreSQL")]

    subgraph DI Detail
        C1["1️⃣ Auth\nJWT decode → User or None"]
        C2["2️⃣ Entity Lookup\nslug/id/username → domain object"]
        C3["3️⃣ Permission Guard\nownership check → 403 or pass"]
        C1 --> C2 --> C3
    end
    C --- DI Detail

Layer 1 — API Routes (app/api/routes/)

Thin HTTP controllers. Each domain has its own sub-router:

Router Domain
authentication.py Register, login
users.py Get/update current user
profiles.py Get profile, follow/unfollow
articles/ Article CRUD, feed, favorites
comments.py Comment CRUD
tags.py Tag listing

Route handlers are intentionally minimal: they accept already-resolved domain objects from dependencies and return typed Pydantic response schemas. No business logic lives here.

Layer 2 — Dependency Injection (app/api/dependencies/)

FastAPI's DI system (Depends(...)) is used extensively to enforce cross-cutting concerns before any handler code runs. Three sub-concerns:

Auth (authentication.py)

# Simplified flow
get_current_user_authorizer(required=True)
  → extract "Bearer <token>" from Authorization header
  → jwt.get_username_from_token()        # decode HS256 JWT
  → UsersRepository.get_user_by_username()  # DB lookup
  → returns User  (or raises 401)

For public endpoints (e.g., listing articles), required=False causes the dependency to return None instead of raising, enabling optional-auth behavior.

Entity resolution

Path parameters (e.g., {slug}, {username}, {comment_id}) are resolved to full domain objects in the DI layer. If the entity doesn't exist, EntityDoesNotExist is caught here and converted to HTTP 404 — before the handler is ever called.

Ownership guards

Ownership checks (e.g., "can this user modify this article?") are separate Depends that compare the authenticated user to the entity's author. A mismatch raises HTTP 403 before the handler executes.

Layer 3 — Services (app/services/)

Pure, stateless utility functions. No I/O, no state.

Service What it does
jwt.py Encode/decode HS256 JWTs; 7-day expiry
security.py bcrypt hash + verify via passlib
articles.py Slug generation (python-slugify), uniqueness check, authz check
authentication.py Email/username uniqueness checks
comments.py Comment ownership check

Layer 4 — Repositories + Queries (app/db/)

All database I/O. Repositories accept an asyncpg.Connection (injected per request) and return typed domain objects — never raw asyncpg.Record rows.

SQL loading strategy — two mechanisms used together:

Mechanism Used for
aiosql Static SQL: loads .sql files from queries/sql/ and exposes them as named Python methods
PyPika Dynamic SQL: article list filtering (tag, author, favorited, pagination) where the WHERE clause changes based on query params

PyPika is a Python query builder that constructs SQL programmatically rather than via string concatenation — safer (parameterized) and easier to read than hand-rolled dynamic SQL. Example: if the ?tag=python query param is present, PyPika appends a JOIN on the tags table; otherwise that JOIN is omitted entirely.

Writes use explicit transactions:

async with connection.transaction():
    await queries.create_article(...)
    await queries.create_tags(...)
    await queries.link_article_tags(...)

5.2 Article List Query (with N+1 read projection)

This is the most complex read path in the system, and contains a known performance concern.

flowchart TD
    A["GET /api/articles?tag=X&author=Y&favorited=Z&limit=N&offset=M"] --> B["Auth DI\n(optional -- User or None)"]
    B --> C["PyPika builds parameterized SQL\nConditionally adds JOINs for tag/author/favorited filters"]
    C --> D["ArticlesRepository\nexecutes query → N article rows"]
    D --> E["For each article row...\n⚠️ N+1 sub-queries per row"]

    subgraph N+1 per row
        E --> F1["SELECT tags WHERE article_id=?"]
        E --> F2["SELECT COUNT favorites WHERE article_id=?"]
        E --> F3["SELECT favorited? WHERE user_id=? AND article_id=?"]
        E --> F4["SELECT author profile + following flag"]
    end

    F1 & F2 & F3 & F4 --> G["Assemble Article domain object\n(with embedded Profile -- denormalized)"]
    G --> H["ArticleListInResponse\n{articles: [...], articlesCount: N}"]

N+1 explained: For a list of 20 articles, the system executes 1 list query + up to 4 sub-queries × 20 articles = ~81 database round trips. This is a known trade-off: read projections are denormalized (each article embeds a full Profile object), which simplifies the response shape but makes bulk reads expensive. This would be the first place to optimize under load (e.g., batch tag/favorites queries, or use a single JOIN-heavy query).

7. Infrastructure Dependencies

PostgreSQL

What it is: A powerful open-source relational database. The gold standard for transactional workloads in Python backends.

Role here: The single source of truth. All application state — users, articles, tags, follows, favorites, comments — lives in PostgreSQL.

Configuration: - Image: postgres:11.5-alpine - Port: 5432 - Connection string: postgresql://postgres:postgres@db/postgres (Docker Compose default) - Data persisted to ./postgres-data volume (survives container restarts) - Connection pool: min=10, max=10 connections (configured in AppSettings)

Schema management: Alembic migrations (see below). The initial migration creates all tables, a updated_at PL/pgSQL trigger, and cascade delete rules.

asyncpg

What it is: A high-performance, fully async PostgreSQL driver for Python. Unlike psycopg2 (the traditional driver), asyncpg is built from the ground up for asyncio — no thread pool workarounds.

Role here: All database I/O goes through asyncpg. A connection pool is created at ASGI startup and attached to app.state.pool. Each request acquires a connection from the pool via a FastAPI dependency, uses it for the request lifetime, then releases it back.

FastAPI / Starlette / Uvicorn

FastAPI is a Python web framework that generates OpenAPI specs automatically from type hints. Starlette is its underlying ASGI toolkit (middleware, routing). Uvicorn is the ASGI server (think: gunicorn, but for async code).

Role here: - FastAPI handles routing, DI, request validation (via Pydantic), and OpenAPI generation. - Uvicorn serves the app in the container: uvicorn --host=0.0.0.0 app.main:app. - Uvicorn's own log output is intercepted and routed through loguru for unified structured logging.

Alembic

What it is: A database schema migration tool for Python. Think: Flyway or Liquibase, but for SQLAlchemy/PostgreSQL ecosystems. Migrations are versioned Python scripts that can be applied incrementally.

Role here: Manages the PostgreSQL schema. The Dockerfile runs alembic upgrade head before starting Uvicorn, so the schema is always current on container start. Migration scripts live in app/db/migrations/.

Note: SQLAlchemy is a dependency of Alembic but is not used for query execution in this project — only as Alembic's migration engine. All actual SQL is handled by asyncpg + aiosql + PyPika.

aiosql + PyPika

aiosql loads .sql files and turns them into named Python callables — keeping SQL in SQL files rather than string literals in Python. PyPika is a programmatic SQL query builder for cases where the query structure is dynamic.

Role here: - Static queries (get user by email, insert article, etc.) → aiosql loading .sql files from app/db/queries/sql/. - Dynamic queries (article list with conditional filters) → PyPika builds the SQL at runtime based on which filter params are present.

PyJWT + passlib/bcrypt

PyJWT encodes and decodes JSON Web Tokens. passlib + bcrypt provide a secure password hashing abstraction.

Role here: - On login/register: bcrypt hashes the plaintext password before storage. bcrypt is intentionally slow (to resist brute-force attacks). - On auth: JWT decode validates the token signature and expiry without a database round-trip. - Token lifetime: 7 days. Algorithm: HS256 (symmetric — the secret_key from settings signs and verifies).

Docker / Docker Compose

Docker packages the app and its dependencies into a reproducible container image. Docker Compose orchestrates multiple containers as a single local environment.

Role here: Two Compose services:

Service Image Port
db postgres:11.5-alpine 5432
app Built from Dockerfile 8000

The app service depends on db. The Dockerfile installs Poetry dependencies, runs Alembic migrations, then starts Uvicorn.

Newman (via npx)

Newman is Postman's headless CLI runner. It executes a Postman collection (a JSON file describing API requests and assertions) from the command line — useful for CI pipelines where you can't use the GUI.

Role here: Exclusively in the test harness. Newman reads Conduit.postman_collection.json, sends the requests in order to the live API, and reports pass/fail counts.

loguru

What it is: A structured logging library for Python that replaces the standard library's logging module with a simpler, more powerful API.

Role here: Unified logging across the application. An InterceptHandler bridges stdlib logging (which FastAPI and Uvicorn use internally) into loguru, so all log output goes through a single configured sink with consistent formatting.

RabbitMQ (⚠️ Unused — Present in Dependencies Only)

RabbitMQ is a message broker implementing AMQP — used to decouple services asynchronously. Think: a queue between a producer and a consumer, where neither needs to know about the other directly.

Status in this codebase: The pika library (Python's AMQP client) appears in the dependency manifest (pyproject.toml/poetry.lock) but there is no RabbitMQ configuration and no usage in source files. This is likely either a leftover dependency or the start of planned async messaging work (e.g., for email notifications or event publishing). Do not treat it as active infrastructure.

9. Key Design Decisions

9.1 DI-First Authorization

FastAPI's dependency system is used for all security enforcement — auth, entity resolution, and ownership guards are all Depends(...) declarations on route functions. This means:

  • Route handlers receive already-validated, already-authorized data.
  • Security cannot be accidentally bypassed by a handler — the framework enforces it.
  • Adding a new protected route means composing existing dependencies, not writing new auth code.

Trade-off: Heavy dependency chains can make route signatures verbose and harder to read for newcomers.

9.2 Denormalized Read Projections (Embedded Profile in Articles)

Article and comment responses embed a full Profile object for the author rather than returning an author ID:

{
  "article": {
    "title": "How to train your dragon",
    "author": {
      "username": "jake",
      "bio": "...",
      "image": "...",
      "following": false
    }
  }
}

This matches the RealWorld spec exactly and means API consumers never need a second request to render an article. The cost is the N+1 sub-query pattern in the repository layer (see §5.2).

9.3 EntityDoesNotExist as a Universal Sentinel

A single exception class (EntityDoesNotExist) is raised by all repository methods when a row is not found. The dependency layer catches this and converts it to HTTP 404. Service-layer uniqueness checks catch it and convert it to False (entity doesn't exist, so the name is available).

This is a control-flow-via-exception pattern. It keeps repository code clean but requires callers to understand the exception contract. Developers should not catch this exception silently in new code without deliberately handling the "not found" case.

9.4 Configuration Hierarchy with @lru_cache

Settings are resolved once at startup:

# Simplified
@lru_cache
def get_app_settings() -> AppSettings:
    env = os.getenv("APP_ENV", "dev")
    return {"dev": DevAppSettings, "prod": ProdAppSettings, "test": TestAppSettings}[env]()

The @lru_cache means the settings object is a process-wide singleton — subsequent calls return the cached instance. This is both a performance optimization and a consistency guarantee (all code sees the same settings object).

Test environment (TestAppSettings) can override database_url, secret_key, etc., enabling isolated test runs without changing production config.

9.5 SQL via Files + aiosql (not an ORM)

The project deliberately avoids SQLAlchemy for query execution. Instead: - Static SQL lives in .sql files — readable by anyone who knows SQL, reviewable in diffs without Python knowledge. - Dynamic queries use PyPika as a structured builder rather than raw string interpolation.

Trade-off: No ORM magic (no lazy loading, no model relationships). Developers write explicit SQL. This is a deliberate choice for transparency and performance control — though it means more boilerplate in the repository layer.

9.6 No Shared Code Between App and Test Suite

The postman/ test suite shares nothing with app/ — no shared models, no shared utilities. It is a pure black-box HTTP consumer. This is the cleanest possible integration test boundary: it proves the API contract works exactly as an external consumer would experience it.

hashed_password is produced by passlib/bcrypt and is never exposed in any API response.

Uniqueness Check Pattern

The service layer performs two read queries before the INSERT:

check_email_is_taken:
    UsersRepository.get_user_by_email(email)
    → EntityDoesNotExist caught → email is available (False)
    → Row returned           → email is taken (True)

check_username_is_taken:
    UsersRepository.get_user_by_username(username)
    → EntityDoesNotExist caught → username is available (False)
    → Row returned           → username is taken (True)

If either check returns True, an HTTP 400 is raised and the INSERT never executes. Note that these checks are not atomic with the INSERT; a TOCTOU race condition is theoretically possible under high concurrency, though the database UNIQUE constraints serve as the final enforcement layer.

Related Data Flow

User Registration: 1. Client POST /api/users {username, email, password} hits the FastAPI ASGI server. 2. CORS middleware passes the request through. 3. Pydantic validates the body as UserInCreate (rejects malformed input with 422). 4. asyncpg connection acquired from pool. 5. check_email_is_taken — UsersRepository.get_user_by_email; EntityDoesNotExist sentinel caught → email available. 6. check_username_is_taken — UsersRepository.get_user_by_username; same sentinel pattern. 7. If either is taken: HTTP 400 {errors: [...]}. 8. security.py: passlib/bcrypt hashes the plaintext password; UserInDB created with the hash. 9. UsersRepository.create_user: INSERT INTO users inside asyncpg transaction; returns new User domain object. 10. jwt.py: create_access_token_for_user encodes HS256 JWT (7-day expiry, secret_key from settings). 11. User domain object + token serialized to UserInResponse.

Internal Query

The repository executes a single SELECT with a LEFT JOIN on the followers table to compute following in one round-trip:

SELECT u.username, u.bio, u.image,
       COUNT(f.follower_id) > 0 AS following
FROM users u
LEFT JOIN followers f
       ON f.following_id = u.id AND f.follower_id = $current_user_id
WHERE u.username = $username

When no authenticated user is present, $current_user_id is NULL and the COUNT evaluates to 0 → following = false.

These checks happen in-process, not in the database, so there is a non-zero TOCTOU window in high-concurrency scenarios; a unique constraint at the database layer serves as the authoritative guard.

This guarantees that whenever the application container starts — whether it's a fresh deployment or a restart — the database schema is automatically brought up to date with the latest migration. asyncpg (Database Driver)

What It Is

asyncpg is a high-performance, fully asynchronous PostgreSQL client library for Python. Unlike traditional database drivers that block the executing thread while waiting for a query to return, asyncpg integrates with Python's asyncio event loop. This means a single application thread can handle many concurrent database queries simultaneously — the thread simply handles other work while waiting for PostgreSQL to respond, rather than sitting idle. This is essential for high-throughput async web applications.

How It's Used Here

asyncpg provides the connection pool that the application uses to talk to PostgreSQL. The pool is initialized once when the application starts up and is managed in app/api/dependencies/database.py. For each incoming HTTP request, FastAPI's dependency injection system acquires one connection from the pool, passes it to the relevant repository class for the duration of the request, and then returns it to the pool when the request is complete.

The Pool and Connection types from asyncpg are used explicitly in the dependency module, making it clear which layer is responsible for connection lifecycle management.
