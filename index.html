What is this?

Spring PetClinic is a web application for managing a veterinary clinic. Staff can register pet owners, add their animals, schedule vet visits, and browse the list of veterinarians and their specialties. Everything is server-rendered HTML, with a small REST/JSON endpoint for the vet list.

It is intentionally a reference application, not a production system. Its real job is to show a clean, idiomatic way to wire together a full-stack Spring Boot application — persistence, validation, caching, internationalisation, and more — all in one coherent codebase you can read in an afternoon.

Why does it exist?

Teams building new Spring Boot services often struggle to answer mundane-but-critical questions: How should I structure my JPA entities? Where does form validation live? How do I cache something properly? PetClinic answers those questions with working, tested code rather than documentation prose.

Concrete example: Sarah joins a team that needs to build a Spring MVC CRUD app backed by PostgreSQL. Instead of assembling boilerplate from scratch, she clones PetClinic, spins up the Postgres profile with docker-compose up, and runs the app in minutes. She studies how OwnerController handles the Post-Redirect-Get pattern, copies the PetValidator approach for her own custom validation, and has a working skeleton for her team by end of day.

High-level walkthrough

The six major pieces:

Piece What it does
Domain model (owner/, vet/) JPA entities — Owner, Pet, Visit, Vet, Specialty. Owner is the aggregate root; saving an owner cascades to its pets and visits.
Repositories Spring Data JPA interfaces. OwnerRepository handles all writes. VetRepository is intentionally read-only.
Controllers Spring MVC HTTP boundary. OwnerController, PetController, and VisitController handle CRUD. VetController serves the paginated HTML view and a JSON/XML REST endpoint.
PetValidator + PetTypeFormatter Custom form glue. The formatter converts a pet-type string from the HTML form into a real PetType domain object. The validator enforces business rules (no duplicate pet names, birth date can't be future-dated).
Caching (CacheConfiguration) The vet list is cached in a JCache region called "vets". Vets change rarely, so this avoids a DB round-trip on every page load.
Database scripts (resources/db/) Dialect-specific DDL and seed SQL for H2 (default/dev), MySQL, and PostgreSQL. Swap profiles to switch databases — no code changes needed.

1. System Overview

Spring PetClinic is a reference Spring Boot web application modeling a veterinary clinic. It is intentionally scoped to be a demonstration platform rather than a production system — its value is the breadth of Spring features it exercises in a single, coherent codebase: JPA persistence, Spring MVC, Bean Validation, caching, i18n, content negotiation, and GraalVM native image support.

The application allows clinic staff to: - Register and search for pet owners - Add and edit pets belonging to an owner - Schedule vet visits for pets - Browse a catalog of veterinarians and their specialties

2. Component Descriptions

2.1 Controllers (HTTP Boundary)

Controllers are the entry point for all HTTP traffic. They follow the Post-Redirect-Get (PRG) pattern: successful mutations always redirect to a GET endpoint, carrying success or error messages in flash attributes (short-lived session entries that survive exactly one redirect).

Controller Path Prefix Responsibility
OwnerController /owners Full CRUD for owners; paginated search with a single-result redirect shortcut
PetController /owners/{ownerId}/pets Pet creation and update scoped under a parent owner
VisitController /owners/{ownerId}/pets/{petId}/visits Visit scheduling scoped under a parent pet
VetController /vets Read-only; serves paginated HTML (/vets.html) and a REST endpoint (/vets) with JSON/XML content negotiation
WelcomeController / Renders the home page
CrashController /oups Deliberately throws a RuntimeException to demonstrate error handling

Flash attributes vs. model attributes: Regular model attributes live only for the current request. Flash attributes are stored in the session and survive one redirect — used here to carry "New Owner Created" confirmation messages from a POST handler to the subsequent GET.

2.2 Repositories (Persistence Boundary)

Repositories are Spring Data JPA interfaces. Spring auto-generates the implementation at startup — you write an interface, Spring provides the SQL.

Repository Extends Notable Methods
OwnerRepository JpaRepository<Owner, Integer> findByLastNameStartingWith(String, Pageable) — paginated last-name search
PetTypeRepository JpaRepository<PetType, Integer> findPetTypes() — sorted JPQL query returning all pet types
VetRepository Repository<Vet, Integer> findAll() and findAll(Pageable) — both annotated @Cacheable("vets")

Why does VetRepository extend Repository instead of JpaRepository? JpaRepository exposes write methods (save, delete, etc.). Since vets are read-only reference data, extending the minimal Repository interface documents and enforces this intent at the type level.

2.3 Supporting Components

Component Purpose
PetValidator Programmatic Spring Validator for Pet form submissions. Checks: name not blank, no duplicate name on the same owner, type not null, birth date not null and not in the future.
PetTypeFormatter Spring MVC Formatter<PetType>. Converts the HTTP string value from a <select> dropdown back into a PetType domain object during form binding. Does a full table scan on every request — see Known Limitations.
CacheConfiguration Declares @EnableCaching and programmatically creates the "vets" JCache region with JMX statistics enabled.
WebConfiguration Registers SessionLocaleResolver and LocaleChangeInterceptor, enabling runtime locale switching via a ?lang=xx query parameter.
PetClinicRuntimeHints Implements RuntimeHintsRegistrar for GraalVM native image compilation. Registers SQL scripts under db/* and message bundles under messages/* as runtime resources, and registers serialization hints for Vets / Vet / Specialty.

Aggregate Roots

The domain is organized around two aggregates:

Owner (aggregate root)
  └── Pet
        └── Visit

Vet (aggregate root, read-only)

All write operations flow through OwnerRepository.save(owner). There are no separate PetRepository.save() or VisitRepository.save() calls — JPA's CascadeType.ALL on the Owner → Pet and Pet → Visit relationships propagates the entire graph in a single save.

This flow applies to both the HTML view (/vets.html) and the REST endpoint (/vets). Both routes share the same JCache region.

JCache (JSR-107) is the standard Java caching API. It defines a vendor-neutral interface; Spring Boot wires in a compliant provider at runtime. Think of it as a named key-value store that lives in the JVM's heap.

6. Infrastructure Dependencies

6.1 Spring Boot

What it is: A Java framework that provides auto-configuration, an embedded HTTP server (Tomcat by default), and opinionated defaults so you can build a production-capable application with minimal boilerplate.

How it's used here: - PetClinicApplication is the @SpringBootApplication entry point — Spring Boot bootstraps JPA, MVC, caching, and database initialization from this single annotation. - Spring Boot Actuator is enabled (management.endpoints.web.exposure.include=*), exposing health, metrics, and info endpoints. - In the Kubernetes deployment, health probes hit /livez and /readyz endpoints exposed by Actuator.

6.2 Spring MVC

What it is: The web framework within Spring for mapping HTTP requests to Java controller methods, binding form data, and rendering responses.

How it's used here: - @Controller classes handle all HTTP routing. - @ModelAttribute methods run before every handler in a controller to pre-load shared data (e.g., the Owner object or available PetType list). - WebDataBinder is configured per-controller to register PetValidator and PetTypeFormatter.

6.3 Spring Data JPA + Hibernate

What it is: Spring Data JPA provides repository interfaces that Spring implements automatically. Hibernate is the JPA provider that translates Java object operations into SQL.

How it's used here: - Schema management is manual (spring.jpa.hibernate.ddl-auto=none) — Hibernate does not auto-create tables. Instead, schema.sql files under resources/db/{dialect}/ are run at startup. - Snake case naming strategy: Java field firstName maps to column first_name automatically via PhysicalNamingStrategySnakeCaseImpl. - Batch fetch size of 16 (hibernate.default_batch_fetch_size=16) reduces the N+1 query risk when loading collections.

6.4 Thymeleaf

What it is: A server-side Java template engine. Controllers pass a model (a Map<String, Object>) to a named template file; Thymeleaf merges them into HTML and returns it in the response body.

How it's used here: - All owner, pet, and visit views are Thymeleaf templates. - Internationalization (i18n) is supported via messages.properties files for 8 locales (en, de, es, fa, ko, pt, ru, tr). The active locale is stored in the HTTP session and switchable via ?lang=xx.

6.5 JCache (JSR-107)

What it is: A standardized Java caching API. Analogous to JDBC for databases — it's an interface layer that lets you swap cache providers (EhCache, Caffeine, Hazelcast) without changing application code.

How it's used here: - CacheConfiguration creates a single named cache region called "vets" with JMX statistics enabled. - @Cacheable("vets") on VetRepository.findAll() tells Spring to intercept the call, check the cache first, and only invoke the real method on a miss.

6.6 GraalVM Native Image (Spring AOT)

What it is: GraalVM can ahead-of-time compile a JVM application into a native binary with faster startup and lower memory — but it requires all reflection, resource loading, and serialization to be declared upfront.

How it's used here: - PetClinicRuntimeHints registers SQL scripts, message bundles, and the Vets/Vet/Specialty serialization graph as hints, ensuring they are included in the native binary.

6.7 Build Tooling

Gradle (primary): Version 9.2.1 via the Gradle wrapper (gradlew). Handles dependency resolution, compilation, testing, and packaging.

Gitpod: Cloud development environment configured via .gitpod.yml. Runs ./mvnw install on workspace init and installs Java and XML VS Code extensions — useful for exploring the codebase without a local JDK.

SQL Script Organization

Each database directory contains two files loaded at startup:

resources/db/
├── h2/
│   ├── schema.sql   ← DDL: CREATE TABLE owners (...), etc.
│   └── data.sql     ← Seed: INSERT INTO owners VALUES (...)
├── mysql/
│   ├── schema.sql
│   └── data.sql
└── postgres/
    ├── schema.sql

This design avoids dialect-specific SQL leaking into application code — all syntax differences (e.g., AUTO_INCREMENT vs SERIAL vs H2 identities) are isolated to these files.

Only the vet catalog is cached. Vets change infrequently (reference data), making them an ideal cache candidate.

Note for engineers: There is no explicit TTL configured in CacheConfiguration.java. In a production context you would want to add cache expiry to ensure stale vet records are eventually refreshed. This is an intentional simplification in the reference app.

9. Key Design Decisions

9.1 Owner as Aggregate Root

Owner is the single save point for the entire Owner → Pet → Visit graph. CascadeType.ALL is set on Owner.pets and Pet.visits, so calling OwnerRepository.save(owner) propagates inserts/updates through the full hierarchy in one transaction.

Implication: You will never find petRepository.save(pet) in this codebase. If you need to add such a repository, be aware that the cascade chain already handles persistence — adding an independent save could cause duplicate insert attempts.

9.2 @MappedSuperclass for the Entity Hierarchy

BaseEntity, NamedEntity, and Person are all @MappedSuperclass — they contribute columns to concrete entity tables but have no tables of their own. For example, Owner maps to the owners table which includes id (from BaseEntity), first_name/last_name (from Person), and address/city/telephone (from Owner).

Trade-off: Simpler queries with no joins to parent tables, at the cost of some column duplication across tables (e.g., first_name and last_name exist in both owners and vets).

9.3 VetRepository Extends Minimal Repository

By extending Spring Data's base Repository<Vet, Integer> (rather than JpaRepository), the interface exposes only the explicitly declared findAll() methods. Write operations like save() and delete() are intentionally absent — the type system enforces the read-only contract.

9.4 Post-Redirect-Get (PRG) Pattern

All successful form submissions redirect to a GET endpoint. This prevents form resubmission on browser refresh and separates the write side (POST handler) from the read side (GET handler). Flash attributes bridge the two requests.

POST /owners/new  →  302 /owners/{id}  →  GET /owners/{id}
       ↑ write           ↑ redirect          ↑ read + show flash msg

9.5 Parallel SQL Scripts per Dialect

Rather than using Hibernate's DDL generation or a migration tool like Flyway, the project maintains three sets of hand-written SQL scripts. This is a deliberate simplicity trade-off: easy to read and modify for a demo, but would not scale to a multi-team production system where schema migrations need to be versioned and audited.

9.6 Content Negotiation on /vets

VetController.showResourcesVetList() uses @ResponseBody and standard Spring MVC content negotiation. The same handler method returns JSON or XML depending on the Accept header — no separate endpoints needed. The Vets class is annotated @XmlRootElement to satisfy JAXB's requirement for a root-level XML element.

10. Known Limitations and Hotspots

Area Issue Location
PetTypeFormatter Calls PetTypeRepository.findAll() (full table scan) on every form binding invocation to resolve the pet type string. For small reference data this is harmless; at scale this is an O(n) hotspot. A simple fix would be to cache pet types similarly to the vet catalog. PetTypeFormatter.parse()
Eager loading of full Owner aggregate VisitController loads the entire Owner → Pets → Visits graph on every @ModelAttribute call (both GET and POST for the visit form). This is two DB round-trips per form submission and grows with the number of visits. VisitController @ModelAttribute
No cache TTL on "vets" The JCache "vets" region has no expiry configured, so vet data will never be refreshed without an application restart unless the cache is manually invalidated via JMX. CacheConfiguration
No authentication All endpoints are publicly accessible. There is no Spring Security configuration. This is intentional for a reference app but must be addressed before any real-world use. Entire application
owner.setId(ownerId) in update handler The owner update flow binds form data to a new Owner object (Spring MVC default) and then manually sets the id from the path variable. If the path variable and form body id diverged (possible in direct API calls), the path variable wins. OwnerController.processUpdateOwnerForm

Content Negotiation Details

Spring MVC content negotiation resolves the response format using the following priority order (standard Spring Boot defaults):

  1. Path extension (not applicable here — no .json or .xml suffix in the path).
  2. Accept header — the primary mechanism for this endpoint.
  3. Default configured media type (typically application/json with Jackson on the classpath).

The Vets class is annotated @XmlRootElement, which enables JAXB marshalling. Jackson handles JSON serialisation through its standard ObjectMapper. Both serialisers are on the classpath in a default Spring Boot project.

1. Application Framework — Spring Boot

What It Is

Spring Boot is a Java framework that removes the boilerplate typically required to build and run Spring applications. 2. Data Access — Spring Data JPA

What It Is

Spring Data JPA is a module that sits on top of the Java Persistence API (JPA) standard and Hibernate (the underlying ORM implementation). It eliminates most boilerplate database code: instead of writing SQL for every query, you define a Java interface that extends a Spring-provided base, and Spring generates the implementation at startup.

How It Is Used Here

All database operations go through Spring Data JPA repositories. The two primary ones are:

  • OwnerRepository — manages Owner entities with pagination support for the owner search page.
  • PetTypeRepository — retrieves pet types (Dog, Cat, etc.) for form dropdowns.

JPA entities (Owner, Pet, PetType, Visit, Vet) map directly to database tables. The spring.jpa.hibernate.ddl-auto=none setting means Hibernate does not create or alter tables automatically — that responsibility belongs to the SQL scripts bundled per database profile.

3. View Rendering — Thymeleaf

What It Is

Thymeleaf is a server-side HTML template engine for Java. Unlike client-side frameworks (React, Vue), Thymeleaf templates are rendered on the server and sent as complete HTML to the browser. Templates are valid HTML files that can be opened directly in a browser even without the server running, which makes them designer-friendly.

How It Is Used Here

Every page the user sees — owner listings, pet forms, vet directories — is a Thymeleaf template. Spring MVC controllers (e.g., OwnerController, PetController) return a view name string such as "owners/ownerDetails" or "pets/createOrUpdatePetForm", and Thymeleaf resolves that to a .html file under src/main/resources/templates/, renders it with model data, and returns the HTML response.

The application supports 8 languages through Spring's internationalization (i18n) mechanism, with messages files for English, German, Spanish, Farsi, Korean, Portuguese, Russian, and Turkish.

4. Caching — JCache (JSR-107)

What It Is

JCache (also called JSR-107) is the standard Java caching API — a vendor-neutral interface that defines how Java applications interact with a cache. Because it is just an API, a real caching provider (such as EhCache or Caffeine) must be present on the classpath to do the actual work. Spring Boot's auto-configuration detects the provider and wires it in automatically.

How It Is Used Here

The veterinarian list is an ideal caching candidate: it rarely changes but is queried on nearly every page load. CacheConfiguration.java annotates itself with @EnableCaching to activate Spring's caching proxy infrastructure, then programmatically creates a cache named "vets" with statistics collection enabled. When VetRepository retrieves the vet list, the result is stored in this cache; subsequent requests are served from memory without hitting the database.

5. Default / Test Database — H2

What It Is

H2 is an in-memory relational database written entirely in Java. "In-memory" means the database lives inside the JVM process — there is no separate database server process to install, start, or connect to. When the application stops, all data is lost. How It Is Used Here

H2 is the default database — it activates when no Spring profile is specified. A developer can clone the repo and run the application immediately without installing any database software. Schema and seed data are loaded from SQL files on the classpath at startup.

9. Cluster Orchestration — Kubernetes

What It Is

Kubernetes (often abbreviated "K8s") is an open-source platform that automates the deployment, scaling, and management of containerized applications. You describe the desired state of your system in YAML manifests (what containers to run, how many replicas, what ports to expose), and Kubernetes continuously works to make the cluster match that description — restarting crashed containers, rescheduling them onto healthy nodes, and routing traffic.

How It Is Used Here

Two manifest files under k8s/ define the full production deployment:

k8s/db.yml — Database layer: - A Deployment that runs one PostgreSQL 18.3 pod. - A Service that gives the database a stable in-cluster DNS name. - A Secret of type servicebinding.io/postgresql that holds connection credentials. This follows the Service Binding Specification, a standard for injecting service credentials into application pods.

k8s/petclinic.yml — Application layer: - A Deployment with 1 replica running the dsyer/petclinic image. - A NodePort Service that maps external port 80 to the container's port 8080. - The DB secret is mounted into the application pod as a projected volume at the path specified by SERVICE_BINDING_ROOT=/bindings. The application reads credentials from this directory at runtime rather than from hardcoded environment variables.
