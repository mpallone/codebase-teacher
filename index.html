What is this?

Conduit is a Medium-style social publishing platform — users register, write articles, follow each other, leave comments, and favorite posts they like. This repo is the backend API for that platform: it handles everything from hashing passwords and issuing login tokens to storing articles, tags, and social relationships in a PostgreSQL database.

It implements the open RealWorld spec, which means the API contract is well-defined and battle-tested. The repo also ships an automated end-to-end test suite that exercises the live API using real HTTP calls.

Why does it exist?

Conduit gives teams a production-grade reference backend they can deploy, fork, or learn from. It's a realistic application — not a toy — so it surfaces the actual tradeoffs you face in a social platform: auth flows, dynamic filtering, denormalized read projections, and migration-managed schema changes.

Concrete example: Maya is a frontend engineer building a new React client against the RealWorld spec. She spins up Conduit locally with docker-compose up, registers a test account via POST /api/users, writes an article, and favorites it — all within minutes. When her UI breaks on the article feed endpoint, she runs the Postman test suite (./postman/run-api-tests.sh) to confirm whether the API itself is healthy before digging into her component code. The suite exits 0, so she knows the bug is in her client. Problem scoped in under a minute.

2.1 app — The ASGI Application Server

The application is a four-layer vertical stack. Each layer has a single, well-defined responsibility.

Layer 1: Routes (app/api/routes/)

Thin HTTP controllers organized by domain. Route handlers do almost no work themselves — they receive already-resolved domain objects from the DI layer, call a repository method, and return a typed Pydantic schema.

Router file Domain Prefix
authentication.py Register, login /api/users
users.py Get/update current user /api/user
profiles.py Profile view, follow/unfollow /api/profiles
articles/articles_resource.py Article CRUD /api/articles
articles/articles_common.py Feed, favorite/unfavorite /api/articles
comments.py Comment CRUD /api/articles/{slug}/comments
tags.py List all tags /api/tags

Layer 2: Dependencies (app/api/dependencies/)

FastAPI's dependency injection system is used as the primary security and validation gate. Dependencies run before the handler executes, so by the time a handler is called, auth is proven and the requested entity is already loaded. Three concerns live here:

  1. Authentication (authentication.py): get_current_user_authorizer(required=True/False) — extracts the Bearer token, decodes the HS256 JWT, and resolves the User from the database. Raises 401 on any failure. Passing required=False allows unauthenticated access to public endpoints while still resolving a user when a token is present.

  2. Entity Resolution (articles.py, comments.py, profiles.py): Converts path parameters (slug, comment ID, username) into domain objects. If the entity doesn't exist, EntityDoesNotExist is caught here and converted to an HTTP 404.

  3. Authorization Guards (articles.py, comments.py): Ownership checks (check_user_can_modify_article, check_user_can_modify_comment) enforce that the currently authenticated user owns the resource. Violations raise 403 before the handler body runs.

Layer 3: Services (app/services/)

Purely stateless utility functions with no direct I/O. These are imported and called by both route handlers and dependencies.

Layer 4: Repositories & Queries (app/db/)

All database I/O is isolated in repository classes. Each repository wraps an injected asyncpg.Connection and returns typed Python domain model objects — raw database rows never escape this layer.

│   ├── base.py           ← BaseRepository(conn: asyncpg.Connection)
│   ├── users.py          ← UsersRepository
│   ├── articles.py       ← ArticlesRepository
│   ├── profiles.py       ← ProfilesRepository
│   ├── comments.py       ← CommentsRepository

├── queries/
│   ├── queries.py        ← aiosql registry (loads all .sql files)

├── migrations/           ← Alembic revision scripts
└── events.py             ← asyncpg pool lifecycle (startup/shutdown)

Query strategy: Static, well-known queries are stored as .sql files and loaded by aiosql into named callable methods. The article listing endpoint requires dynamic WHERE/JOIN construction based on optional filter parameters (tag, author, favorited); for this case, PyPika (a Python SQL query builder) is used to programmatically compose parameterized SQL. This avoids string concatenation while keeping the dynamic logic explicit.

app/main.py — App Factory & Lifecycle

  • Creates the FastAPI application instance.
  • Registers CORS middleware (permissive: all origins/methods/headers).
  • Wires custom error handlers: http_error_handler normalizes HTTPException → {"errors": [...]}, and http422_error_handler reformats Pydantic validation errors to the same envelope shape.
  • Registers startup/shutdown handlers from events.py to create and teardown the asyncpg connection pool.

5. Infrastructure Dependencies

PostgreSQL

PostgreSQL is the sole persistent data store. The schema is owned and versioned by Alembic.

  • Connection management: asyncpg maintains a connection pool (min 10, max 10 connections) attached to app.state.pool. Connections are acquired per-request via the get_connection dependency and injected into repository constructors. The pool is created on ASGI startup and closed on shutdown (app/db/events.py).
  • Schema highlights: A PL/pgSQL trigger automatically sets updated_at = NOW() on article updates. Favorites and followers use ON CONFLICT DO NOTHING for idempotent writes. Article and comment deletions cascade.
  • Migrations: Alembic runs in "online mode" only (no offline/SQL generation). The single initial revision creates the complete schema. The Docker entrypoint runs alembic upgrade head before starting the server.
Tables: users, articles, tags, article_tags, favorites, followers, comments

FastAPI / Starlette

FastAPI is the HTTP framework. Key integrations: - Dependency Injection: FastAPI's Depends() system is used for auth, DB connection acquisition, entity loading, and permission guards. Dependencies compose hierarchically — auth resolves before entity lookup, which resolves before the handler. - Pydantic: Request bodies are validated as Pydantic models. Response objects are serialized through Pydantic schema classes that enforce the RealWorld envelope format. - OpenAPI: Auto-generated at /openapi.json; browsable UI at /docs and /redoc.

asyncpg

asyncpg is a high-performance, purely async PostgreSQL driver. It does not use SQLAlchemy's ORM or core — queries are raw SQL strings (loaded via aiosql) executed directly against asyncpg.Connection objects. SQLAlchemy appears only as a migration-runner dependency for Alembic.

aiosql

aiosql loads .sql files from app/db/queries/sql/ and binds them to named Python methods at startup. This keeps SQL out of Python source files while retaining full control over query text. For example, queries.get_user_by_email(conn, email=email) executes the contents of get_user_by_email.sql.

PyPika

PyPika is a Python query builder (not an ORM). It is used exclusively for the article listing endpoint, where the WHERE/JOIN clauses vary based on which filter parameters are provided. The result is a parameterized SQL string passed to asyncpg.

PyJWT + passlib/bcrypt

  • PyJWT signs and verifies HS256 access tokens. Tokens carry a username claim and expire after 7 days. The signing secret comes from settings.secret_key.
  • passlib/bcrypt handles password hashing. The CryptContext in security.py is a thin wrapper used by UserInDB during creation and by the login handler during verification.

Alembic

Alembic manages schema migrations against PostgreSQL. It is invoked as a CLI command (alembic upgrade head) rather than programmatically. SQLAlchemy is used only to provide a synchronous engine for Alembic's migration runner; it plays no role in the application's data access path.

Newman (npx)

Newman is Postman's CLI collection runner. It is invoked via npx (no global install required) by the test harness shell script. It parses the Postman collection JSON, executes requests in sequence, chains values between requests (e.g., storing the JWT from login and injecting it into subsequent authenticated requests), and evaluates per-request assertion scripts.

Uvicorn

Uvicorn is the ASGI server that hosts the FastAPI application. It is started by the Docker CMD after Alembic migrations complete. Uvicorn's internal loggers (uvicorn.asgi, uvicorn.access) are intercepted by InterceptHandler and routed through Loguru for unified structured logging.

Docker / Docker Compose

7. Key Design Decisions

Decision 1: Dependency Injection as a Security Gate

FastAPI's Depends() system is used not just for convenience (database connection injection) but as the primary security enforcement mechanism. Auth resolution, entity loading, and ownership checks are all declared as dependencies on route function signatures. This means:

  • A handler function cannot execute unless auth has been verified and the entity exists.
  • Security logic is testable in isolation from handler logic.
  • Adding a new protected endpoint is a one-line annotation (Depends(get_article_by_slug_from_path)).

The tradeoff is that the dependency graph becomes implicit — to understand what a handler receives, you must trace through multiple layers of Depends.

Decision 2: Repository Layer as the DB Boundary

All asyncpg calls live in repository classes. Domain model objects (not raw asyncpg.Record dicts) are returned. This means:

  • SQL schema changes are isolated to repositories/ and queries/sql/.
  • Type safety is enforced at the boundary — callers always receive typed Python objects.
  • The N+1 query pattern is a consequence of this design: the repository is responsible for hydrating the full read projection before returning.

Decision 3: Static SQL Files + PyPika for Dynamic Queries

Most queries are static .sql files managed by aiosql. This keeps SQL readable, version-controllable, and out of Python logic. The one exception is article listing, which requires dynamic filtering — here, PyPika generates parameterized SQL programmatically rather than through unsafe string concatenation. This hybrid approach gives the best of both worlds for this codebase's query patterns.

Decision 4: HS256 JWTs with 7-Day Expiry, No Refresh

Tokens are stateless HS256 JWTs — no token store, no revocation list. The signing secret is a single environment variable. Consequences:

  • Simpler: no token management infrastructure required.
  • Less secure: a compromised token is valid for up to 7 days and cannot be revoked without rotating the secret key (which invalidates all tokens).
  • There is no refresh token mechanism. This is appropriate for a demo platform but would need redesign for a production system with real security requirements.

Decision 5: ON CONFLICT DO NOTHING for Idempotent Writes

Favorites (INSERT INTO favorites) and follows (INSERT INTO followers) use PostgreSQL's ON CONFLICT DO NOTHING. This means repeated POST /favorite or POST /follow calls are safe and return the correct state without error — client-side retry is free. The same pattern applies to tag creation (INSERT INTO tags).

Decision 6: Configuration via Hierarchical Settings with @lru_cache

Settings are resolved once at startup and cached for the process lifetime. The APP_ENV variable selects between Dev, Prod, and Test settings classes, each of which can override any default. This pattern means:

  • No configuration is read at import time — only when get_app_settings() is first called.
  • Tests can override settings by substituting a TestAppSettings instance.
  • There is no runtime configuration reload.

Decision 7: Unified Logging via Loguru + InterceptHandler

Python's stdlib logging and Uvicorn's internal loggers are all intercepted and routed through loguru. This produces a single, consistently formatted log stream regardless of which library emits the message. The tradeoff is a custom bridge class (InterceptHandler) that must be maintained.

  2. asyncpg connection acquired from pool.

  4. PyPika (via queries/tables.py) dynamically builds a parameterized SQL SELECT:
  5. If tag is present: adds JOIN article_tags at ON articles.id = at.article_id JOIN tags t ON at.tag_id = t.id WHERE t.tag = $tag.

  7. If favorited is present: adds JOIN favorites f ON articles.id = f.article_id JOIN users fu ON f.user_id = fu.id WHERE fu.username = $favorited.

  9. ArticlesRepository executes the dynamically assembled query via asyncpg.
  10. For each returned row, _get_article_from_db_record fires sub-queries:
  11. SELECT tag FROM tags WHERE article_id=? — tag list
  12. SELECT COUNT(*) FROM favorites WHERE article_id=? — favorites count
  13. SELECT 1 FROM favorites WHERE article_id=? AND user_id=? — favorited by current user (skipped if unauthenticated)
  14. SELECT … FROM users LEFT JOIN followers … WHERE username=? — author profile with following flag
  15. Articles assembled into domain objects and serialized into ArticleListInResponse.

  • articlesCount is the total count of matching articles independent of limit/offset. Use it to compute total pages: ceil(articlesCount / limit).
  • The N+1 sub-query pattern is a known design characteristic of this codebase. For large result sets, prefer smaller limit values.
  • Results are ordered by createdAt descending (most recent first) unless the underlying query is modified.

This endpoint implements the User Registration flow.

  1. CORS middleware passes the request through.
  2. Pydantic validates the request body as UserInCreate. Missing or malformed fields → 422.
  3. asyncpg connection acquired from pool.
  4. Uniqueness check — email: UsersRepository.get_user_by_email(email). If EntityDoesNotExist is raised, email is available; if a user is returned, email is taken. Result stored as boolean.

  10. User domain object + token serialized to UserInResponse.

INSERT INTO users (username, email, hashed_password, bio, image)
VALUES ($1, $2, $3, NULL, NULL)
RETURNING id, username, email, hashed_password, bio, image;

Security Notes

  • Passwords are hashed with bcrypt (via passlib) before any database write. The plaintext password exists only in memory during the request lifecycle.
  • The uniqueness checks use the EntityDoesNotExist sentinel pattern (catching the absence of a record rather than querying for existence). This means two concurrent registrations with the same email could theoretically both pass the check; a database-level UNIQUE constraint on users.email and users.username serves as the final consistency guarantee.
  • Newly registered users receive a token immediately — no email verification step is implemented.

Password handling: The plaintext password is passed through security.py → passlib/bcrypt before being written to the database. The raw value is never logged or stored.

Response — 200 OK

Returns the updated user object. Note that token is included but is not rotated; it is the same JWT that was used to authenticate the request.

{
  "user": {
    "email": "newemail@example.com",
    "username": "alice",
    "token": "eyJhbGciOiJIUzI1NiJ9...",
    "bio": "Updated biography.",
    "image": "https://example.com/new-avatar.jpg"
  }
}

Error Responses

Status Condition
400 Bad Request The new email is already registered to a different account, or the new username is already taken. Response body contains a descriptive errors object.
401 Unauthorized Missing, malformed, or expired JWT.
422 Unprocessable Entity Request body fails Pydantic validation (e.g. email field is not a valid email string).
