What is this?

Spring PetClinic is a web application for managing a veterinary clinic. Front-desk staff can register pet owners, add their animals, schedule clinic visits, and browse the list of on-staff veterinarians — all through a straightforward browser UI. A lightweight REST endpoint also exposes vet data as JSON or XML for any integrations that need it.

It doubles as a living reference application for the Spring ecosystem, so the code is intentionally kept clean and illustrative. If you're here to ship a feature, this context is just background noise — the app is real and fully functional.

Why does it exist?

The application solves a real administrative problem: clinics need a single place to track who owns which pets, what conditions were treated, and which vets have which specialties. Without it, that information lives in scattered paper records or spreadsheets.

Concrete example: Maria brings her cat Biscuit in for the third time this month. The receptionist searches "Garcia" in the owners list, pulls up Maria's profile in one click, sees Biscuit's full visit history on the same page, and books today's appointment — all in under a minute. No digging through files.

Beyond the clinic use case, the project is widely used as a teaching tool. If you're onboarding to learn how the Spring team structures a production-quality app, you're in exactly the right place.

High-level walkthrough

Major pieces:

Piece Job

VetController Serves the paginated vet list (HTML) and the /vets REST endpoint (JSON/XML)
OwnerRepository The single save path for owners, pets, and visits — they all cascade through here
VetRepository Read-only vet queries; results are cached so the DB isn't hit on every page load

How a request flows — booking a visit:

sequenceDiagram
    Browser->>VisitController: POST /owners/42/pets/7/visits/new
    VisitController->>OwnerRepository: findById(42) -- loads Owner + Pets + Visits (eager)
    VisitController->>VisitController: validate description (@NotBlank)
    VisitController->>OwnerRepository: save(owner) -- cascades INSERT into visits
    OwnerRepository->>DB: INSERT INTO visits
    VisitController->>Browser: 302 → /owners/42

The key design rule to internalize early: there is no PetRepository or VisitRepository. Every write to a pet or visit flows through OwnerRepository.save(owner) via JPA cascade. Once you see that, the controller code clicks into place immediately.

The vet side is simpler — it's read-only and cached. VetRepository.findAll() is wrapped in a "vets" JCache entry, so repeated calls to /vets.html hit memory, not the database.

Running it locally takes one command — H2 runs embedded by default, no external database needed:

./mvnw spring-boot:run
# App is live at http://localhost:8080

To run against MySQL or PostgreSQL instead, spin up the provided Docker Compose file and activate the matching Spring profile (--spring.profiles.active=mysql or postgres).

1. System Overview

Spring PetClinic is a full-stack veterinary clinic management application built on Spring Boot. It handles:

  • Owner & pet management — CRUD for clinic owners and their pets
  • Visit recording — logging clinic visits against a specific pet
  • Vet directory — browsing veterinarians and their specialties

It is also an official reference application for Spring ecosystem best practices: JPA persistence, MVC, caching, i18n, and GraalVM native image compilation.

3. Component Descriptions

3.1 Owner Domain (owner/)

This is the core of the application. Owner is the aggregate root: all writes to pets and visits must go through OwnerRepository.save(owner).

Class Type Responsibility
Owner @Entity Aggregate root. Holds List<Pet> with CascadeType.ALL and FetchType.EAGER.
Pet @Entity A pet belonging to an owner. Holds a PetType and a Set<Visit> with CascadeType.ALL, FetchType.EAGER.
PetType @Entity Reference/lookup entity (e.g., cat, dog). Read from the types table.
Visit @Entity A clinic visit record. date defaults to LocalDate.now() on construction.
OwnerRepository Spring Data JPA Supports paginated last-name search and full CRUD on the Owner aggregate.
PetTypeRepository Spring Data JPA Sorted lookup of all PetType rows. Consumed by PetTypeFormatter and controllers.
OwnerController @Controller CRUD routes for owners: create, search, view, update (/owners/**).
PetController @Controller Create/update pets scoped to an owner (/owners/{ownerId}/pets/**).
VisitController @Controller Records new visits against a specific pet (/owners/{id}/pets/{id}/visits/new).
PetValidator Validator Domain-level custom validation: name not blank, type required for new pets, birthDate not null.
PetTypeFormatter Formatter<PetType> Spring MVC Formatter that converts a form string like "cat" into a managed PetType JPA entity on every form bind. Queries PetTypeRepository on every call.

3.2 Vet Domain (vet/)

A read-heavy subdomain. All reads are cached, and the repository is intentionally read-only.

Class Type Responsibility
Vet @Entity Extends Person. Has a many-to-many relationship with Specialty via the vet_specialties join table. JAXB-annotated for XML output.
Specialty @Entity Named lookup entity (specialties table).
Vets DTO @XmlRootElement wrapper around List<Vet>. Used as the body for the REST endpoint — Jackson handles JSON, JAXB handles XML.
VetRepository Repository<Vet, Integer> Extends base Repository (not JpaRepository) — intentionally exposes only read operations. Both findAll() overloads are @Cacheable("vets").
VetController @Controller Serves /vets.html (paginated Thymeleaf view) and GET /vets (JSON/XML REST response).

3.3 System Infrastructure (system/)

Class Responsibility
CacheConfiguration Registers a JCache "vets" cache via JCacheManagerCustomizer with JMX statistics enabled. Spring's @EnableCaching activates the abstraction.
WebConfiguration Configures a SessionLocaleResolver and a LocaleChangeInterceptor so users can switch language via the ?lang= query parameter.
WelcomeController Routes GET / to the welcome Thymeleaf template.
CrashController GET /oups deliberately throws a RuntimeException to demonstrate Spring MVC error handling — useful for testing error pages.

3.4 Shared Model Base Classes (model/)

All JPA entities share a common inheritance hierarchy. These are @MappedSuperclass, meaning they contribute fields to subclass tables with no tables of their own.

BaseEntity          → id (Integer), isNew() → true if id == null
  └── NamedEntity   → + name (String)
        ├── PetType
        └── Specialty
  └── Person        → + firstName, lastName
        ├── Owner
        └── Vet

3.5 AOT / Native Image Support (PetClinicRuntimeHints)

GraalVM native image compilation requires upfront knowledge of reflection, resources, and serialization — things that are normally resolved at runtime in the JVM. PetClinicRuntimeHints implements RuntimeHintsRegistrar and is referenced via @ImportRuntimeHints on the main application class. It registers:

  • Classpath resources (e.g., message bundles)
  • Domain classes that need to be serializable at native compile time

7. Infrastructure Dependencies

Spring Boot

What it is: A Java framework that wraps the Spring ecosystem with auto-configuration and an embedded web server (Tomcat by default). You write application logic; Spring Boot wires everything together.

The entire application bootstraps from PetClinicApplication.java annotated with @SpringBootApplication. Key properties:

spring.jpa.open-in-view=false          # Prevents lazy-load leaks into view layer
management.endpoints.web.exposure.include=*  # Exposes all Actuator endpoints
spring.web.resources.cache.cachecontrol.max-age=12h

Spring MVC

What it is: The web layer of Spring. Controllers handle HTTP requests; Model objects carry data to views; BindingResult captures validation errors.

Controllers use @InitBinder to block id from being bound from form data (security hardening against mass-assignment). @ModelAttribute methods pre-load common data (e.g., the current owner, pet types list) before each controller method runs.

Spring Data JPA + Hibernate

What it is: Spring Data JPA generates repository implementations from interfaces (like OwnerRepository). Hibernate is the underlying ORM (Object-Relational Mapper) that translates Java objects to SQL.

Key configuration:

spring.jpa.hibernate.ddl-auto=none     # Schema managed by SQL scripts, not Hibernate
spring.jpa.hibernate.naming.physical-strategy=PhysicalNamingStrategySnakeCaseImpl
                                       # Java camelCase → SQL snake_case automatically
spring.jpa.properties.hibernate.default_batch_fetch_size=16
                                       # Batches secondary collection fetches in groups of 16

JCache (JSR-107)

What it is: A standard Java caching API. Think of it as a Map that sits in front of your database. Spring's @Cacheable annotation intercepts method calls and returns cached results instead of hitting the DB every time.

The "vets" cache is configured in CacheConfiguration with JMX statistics enabled, meaning you can inspect cache hit/miss rates via JMX tools (e.g., JConsole, VisualVM) without any additional setup. Configuration:

// CacheConfiguration.java
config.setStatisticsEnabled(true);
manager.createCache("vets", config);

Only the vet data is cached because it is read-heavy and changes infrequently.

Thymeleaf

What it is: A Java server-side HTML template engine. Templates are valid HTML files that can be opened in a browser directly (useful for designers). Spring MVC resolves view names like "owners/ownerDetails" to resources/templates/owners/ownerDetails.html.

Configuration: spring.thymeleaf.mode=HTML

Jakarta Bean Validation

What it is: A standard API (formerly javax.validation) for declaring constraints on Java objects via annotations. Spring MVC triggers validation automatically when a method parameter is annotated with @Valid.

Used annotations in this codebase: - @NotBlank — on owner/vet name fields - @Pattern(regexp = "...") — on telephone number - Custom PetValidator — checks pet name uniqueness within an owner's pet collection and enforces type selection only for new pets

JAXB + Jackson

What they are: JAXB (Java Architecture for XML Binding) serializes Java objects to/from XML. Jackson does the same for JSON. Both are integrated into Spring MVC's content negotiation.

The /vets REST endpoint returns the same Vets wrapper object; Spring MVC inspects the Accept header and delegates to JAXB (for application/xml) or Jackson (for application/json). Vet is annotated with @XmlRootElement and @XmlElement to control the XML structure.

GraalVM Native Image (AOT)

What it is: GraalVM can compile a Spring Boot app into a standalone native binary (no JVM required at runtime), resulting in near-instant startup and lower memory usage. It requires all reflection, resource loading, and serialization to be declared at build time.

PetClinicRuntimeHints implements RuntimeHintsRegistrar and is picked up by Spring AOT at build time to register these declarations. The annotation @ImportRuntimeHints(PetClinicRuntimeHints.class) on the main application class activates it.

10. Key Design Decisions

Decision 1 — Owner as Sole Aggregate Root

What: There is no PetRepository or VisitRepository. All writes to Pet and Visit are done by calling OwnerRepository.save(owner), which triggers CascadeType.ALL to cascade saves down the object graph.

Why: This enforces DDD (Domain-Driven Design) aggregate boundaries. Pet and Visit cannot exist independently of an Owner; making them inaccessible via their own repositories prevents inconsistent state where a Pet exists without an Owner.

Trade-off: Every save loads and re-saves the entire owner aggregate. This is acceptable for a single-clinic application with modest data volumes, but would be an N+1 and over-write concern at scale.

// Correct: write through the aggregate root
OwnerRepository.save(owner);  ✅

// Does not exist — enforces aggregate boundary
PetRepository.save(pet);      ❌ (no such interface)

Decision 2 — Read-Only VetRepository

What: VetRepository extends Spring Data's base Repository<Vet, Integer> rather than the full JpaRepository. This intentionally exposes only the findAll() methods declared in the interface.

Why: The vet subdomain is purely read-heavy — vets are not created or updated through this UI. Making the repository read-only prevents accidental mutations and signals intent to future developers.

Comparison:

// VetRepository — intentionally limited
interface VetRepository extends Repository<Vet, Integer> {
    @Cacheable("vets")
    List<Vet> findAll();
}

// OwnerRepository — full write capability
interface OwnerRepository extends JpaRepository<Owner, Integer> {
    Page<Owner> findByLastNameStartingWith(String lastName, Pageable pageable);
}

Decision 3 — PetTypeFormatter as DB-Querying Form Bridge

What: PetTypeFormatter implements Spring MVC's Formatter<PetType> interface. On every form POST that includes a pet type string, it queries PetTypeRepository.findByName() to resolve the string into a managed JPA entity.

Why: HTML forms submit only strings. The JPA layer needs managed entity objects with database-assigned IDs. The Formatter is Spring MVC's clean extension point for this bidirectional conversion. Querying the DB on every parse ensures the entity is always fresh and managed within the current persistence context.

Trade-off: One extra SELECT per form submission. Acceptable for low traffic, but could be replaced with a lookup cache for high-volume scenarios.

Decision 4 — EAGER Fetching Throughout

What: Both Owner → pets (FetchType.EAGER) and Pet → visits (FetchType.EAGER) are eagerly loaded.

Why: Simplifies controller code — no need for explicit join fetches or lazy-load workarounds. spring.jpa.open-in-view=false is set (which closes the persistence context before the view renders), so EAGER fetching is necessary to avoid LazyInitializationException.

Trade-off: A query for any Owner always fetches the full tree: all pets and all visits. For an owner with many pets and a long visit history, this is wasteful. hibernate.default_batch_fetch_size=16 mitigates some N+1 effects on collections, but this would need pagination or lazy loading in a production-grade system.

Decision 5 — @InitBinder Blocking the id Field

What: Both OwnerController and VisitController call binder.setDisallowedFields("id") in an @InitBinder method.

Why: The app is a reference implementation demonstrating Spring i18n best practices. Session-based locale means the user's language preference persists across page navigations without needing a URL prefix or cookie.

It provides auto-configuration (sensible defaults wired from the classpath), an embedded servlet container (Tomcat by default — no separate server installation required), production-ready health and metrics endpoints via Actuator, and an opinionated project structure.

How it's used here: PetClinicApplication.java is the application entry point, annotated with @SpringBootApplication. Everything — web controllers, JPA repositories, caching, and health endpoints — is wired through Spring Boot's auto-configuration. The application listens on port 8080.

Spring Data JPA / Hibernate

What it is: JPA (Java Persistence API) is the Java standard for object-relational mapping (ORM). Hibernate is the most widely used JPA implementation — it translates operations on Java objects into SQL. Spring Data JPA sits on top of Hibernate and provides repository interfaces (e.g., JpaRepository) that automatically implement common database operations (find, save, delete, count) with minimal code.

How it's used here: Every domain model (Owner, Pet, PetType, Visit, Vet) is annotated with JPA annotations such as @Entity, @Table, and @Column. Repositories like OwnerRepository and PetTypeRepository extend JpaRepository, getting full CRUD and pagination support without writing SQL.

Configuration:

Property Value Purpose
spring.jpa.hibernate.ddl-auto none Hibernate does not generate or modify schema; DDL is managed via SQL scripts
spring.jpa.hibernate.naming.physical-strategy PhysicalNamingStrategySnakeCaseImpl Java camelCase field names map to snake_case database column names automatically
spring.jpa.properties.hibernate.default_batch_fetch_size 16 Hibernate fetches up to 16 lazy associations in a single SQL IN clause, reducing N+1 query problems

Thymeleaf

What it is: Thymeleaf is a server-side HTML template engine for Java. It processes .html template files on the server, substitutes dynamic data (e.g., a list of owners from the database), and returns fully rendered HTML to the browser. Unlike JSPs, Thymeleaf templates are valid HTML files that can be opened directly in a browser for design work.

How it's used here: Controllers such as OwnerController and PetController return a template name string (e.g., "owners/ownerDetails", "pets/createOrUpdatePetForm"). Spring Boot's Thymeleaf auto-configuration resolves these names to files under src/main/resources/templates/ and renders them as HTTP responses.

Configuration:

Property Value Purpose
spring.thymeleaf.mode HTML Thymeleaf operates in standard HTML5 mode (lenient parsing, not strict XML)

JCache (JSR-107)

What it is: JCache is the standard Java caching API (JSR-107). It defines a vendor-neutral interface for in-process (in-memory) caching — storing frequently accessed data in memory to avoid repeated expensive operations like database queries. Spring Boot can auto-configure a compliant caching provider (e.g., EhCache or Caffeine) at runtime when one is present on the classpath.

How it's used here: CacheConfiguration.java uses JCacheManagerCustomizer to programmatically create a cache named "vets" with statistics enabled. The @EnableCaching annotation on the application activates Spring's caching abstraction. When the vet list is requested, the result is served from the cache rather than hitting the database on every request.

H2 (Embedded — Default)

What it is: H2 is a lightweight relational database written entirely in Java. It runs inside the application's JVM process with no external server or installation required. Data is stored in memory and is lost when the application stops. It is ideal for development and automated testing.

How it's used here: H2 is the default database — active when no Spring profile is set. The schema is initialised from classpath:db/h2/schema.sql and seeded with sample data from classpath:db/h2/data.sql on every startup.

Configuration (application.properties):

Property Value
database h2
spring.sql.init.schema-locations classpath*:db/h2/schema.sql
spring.sql.init.data-locations classpath*:db/h2/data.sql
spring.jpa.hibernate.ddl-auto none (schema managed by SQL scripts, not Hibernate)

⚠️ H2 is not suitable for production. All data is lost on restart. Use MySQL or PostgreSQL for persistent workloads.

Docker

What it is: Docker packages an application and all its dependencies into a container — a lightweight, isolated, portable unit that runs identically regardless of where it is deployed (a developer's laptop, a CI server, a cloud VM). Containers share the host OS kernel but are otherwise isolated from each other.

How it's used here: docker-compose.yml defines two services for local development so developers can run MySQL or PostgreSQL without installing either database natively. Additionally, .gitpod.yml references .devcontainer/Dockerfile to define the container image used inside the Gitpod cloud workspace.

Kubernetes

What it is: Kubernetes (K8s) is a container orchestration platform. It manages the deployment, scaling, self-healing, and networking of containerised applications across a cluster of machines. Rather than running containers manually with docker run, you declare the desired state in YAML manifests and Kubernetes continuously reconciles the actual state to match.

How it's used here: Two manifest files live in k8s/:

  • k8s/db.yml — deploys a PostgreSQL database with its credentials stored in a Kubernetes Secret.
  • k8s/petclinic.yml — deploys the PetClinic application and exposes it via a NodePort Service. The app receives database credentials via a Service Binding volume mount (the demo-db Secret is mounted into the container at /bindings).
