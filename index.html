What is this?

Conduit is a fully featured social publishing backend — think a stripped-down Medium. It lets users register, write and discover articles, follow authors, favorite posts, and leave comments. The whole thing is built as a clean, async Python web service and implements the open RealWorld specification, which defines a shared API contract so different technology stacks can be compared apples-to-apples.

If you're here to add a feature, fix a bug, or understand how a well-structured async Python API hangs together, you're in the right place.

Why does it exist?

"Todo apps" don't tell you much about how a real backend handles auth, pagination, relational data, or authorization rules. Conduit solves that by being a realistic, deployable application — not a toy. It's the reference implementation teams reach for when they want proven async Python API patterns, or when they need a working social platform API to build a front-end against.

Concrete example: Priya is building a React front-end for a content platform. She needs a real API with JWT auth, article CRUD, and a social follow graph — but she doesn't want to write the backend from scratch. She clones Conduit, runs docker-compose up, and has a fully functional API at localhost:8000/api in under two minutes. She registers a user, publishes an article, follows an author, and her front-end is talking to real data before lunch.

High-level walkthrough

The five pieces you need to know:

Piece What it does
app/api/routes/ Thin HTTP controllers — one file per domain (articles, users, profiles, comments, tags)
app/api/dependencies/ Resolves domain objects from each request: decodes JWT into a User, slug into an Article, enforces ownership
app/services/ Pure business logic, no I/O — password hashing, JWT creation, slug generation, authorization checks
app/db/repositories/ All database reads and writes; each repository assembles full domain objects from raw SQL results
postman/ Black-box integration tests that drive the live API over HTTP using Newman

How a typical request flows end-to-end:

A few things worth knowing up front: all SQL lives in .sql files under app/db/queries/sql/ and is loaded as callable Python functions — queries are readable and version-controlled, with no ORM at runtime. Domain models are plain Pydantic objects assembled by repositories, not ORM entities. The postman/ folder is entirely independent — it talks to the API purely over HTTP and retargets environments by setting a single APIURL variable.

1. System Overview

Conduit is a production-grade REST API implementing the RealWorld specification — effectively a Medium-clone backend. It is written in async Python using FastAPI, backed by PostgreSQL, and structured as a reference implementation for modern Python web service patterns.

The system has two top-level modules:

Module Role
app The FastAPI backend — HTTP surface, business logic, and data access
postman A black-box integration test suite driven by Newman (Postman's CLI runner)

These modules communicate only over HTTP. The test suite has no shared code with the application — it interacts exactly as an external client would.

Core capabilities: - User registration, login, and JWT-based authentication - Article authoring with slugs, tags, and full CRUD - Personalized social feed (articles from followed authors) - Favorites and follow/unfollow social graph - Comment threads on articles

3. Component Descriptions

3.1 API Layer (Routes)

Location: app/api/routes/

The route layer is intentionally thin — handlers contain no business logic. Each handler's only responsibilities are:

  1. Declare what it needs via FastAPI's Depends(...) injection
  2. Call repository or service methods
  3. Serialize the result into a response schema
app/api/routes/
├── api.py                   ← Top-level router; aggregates all sub-routers
├── authentication.py        ← POST /users, POST /users/login
├── users.py                 ← GET /user, PUT /user
├── profiles.py              ← GET /profiles/{username}, POST/DELETE /follow
├── articles/
│   ├── articles_resource.py ← CRUD: GET/POST /articles, GET/PUT/DELETE /articles/{slug}
│   └── articles_common.py   ← Feed: GET /feed, POST/DELETE /{slug}/favorite
├── comments.py              ← GET/POST/DELETE /articles/{slug}/comments
└── tags.py                  ← GET /tags

The API prefix is /api, so all endpoints are reachable at /api/.... Interactive docs are served at /docs (Swagger UI) and /redoc.

Example handler — article creation (simplified):

@router.post("/", response_model=ArticleInResponse, status_code=HTTP_201_CREATED)
async def create_new_article(
    article_create: ArticleInCreate = Body(..., embed=True, alias="article"),
    user: User = Depends(get_current_user_authorizer()),
    articles_repo: ArticlesRepository = Depends(get_repository(ArticlesRepository)),
) -> ArticleInResponse:
    slug = get_slug_for_article(article_create.title)
    await check_article_exists(articles_repo, slug)
    article = await articles_repo.create_article(article=article_create, slug=slug, author=user)
    return ArticleInResponse(article=article)

The handler declares its dependencies (User, ArticlesRepository) but never constructs them — FastAPI resolves them automatically.

3.2 Dependency Injection Layer

Location: app/api/dependencies/

FastAPI's dependency injection system (similar in concept to Spring's @Autowired or .NET's constructor injection) resolves declared Depends(...) arguments before the route handler runs. This is where authentication, authorization, and object resolution happen.

app/api/dependencies/
├── database.py         ← get_repository(RepoType) — scopes a DB connection per request
├── authentication.py   ← get_current_user_authorizer() — JWT → User object
├── articles.py         ← get_article_by_slug_from_path — slug → Article (404 on miss)
│                          check_article_modification_permissions — ownership guard (403)
├── comments.py         ← get_comment_by_id_from_path — id → Comment
│                          check_comment_modification_permissions — ownership guard (403)
└── profiles.py         ← get_profile_by_username_from_path — username → Profile

get_repository(RepoType) — the DB connection factory:

This is the central mechanism for database access. It acquires a connection from the asyncpg pool (stored on app.state.pool) and wraps it in the appropriate repository class. The connection is scoped to the lifetime of a single HTTP request.

def get_repository(repo_type: Type[BaseRepository]) -> Callable:
    async def _get_repo(
        pool: Pool = Depends(get_pool),
    ) -> BaseRepository:
        async with pool.acquire() as connection:
            yield repo_type(connection)
    return _get_repo

get_current_user_authorizer(required: bool) — two modes:

required Missing/invalid token Valid token
True Raises HTTP 403 Returns User
False Returns None (anonymous) Returns User

This pattern is used so that listing articles (GET /api/articles) can show viewer-relative data (e.g., favorited: true) for authenticated users while still serving anonymous requests.

3.3 Service Layer

Location: app/services/

The service layer contains pure Python functions — no I/O, no database calls, no HTTP concerns. Each function takes domain objects and returns a result or raises an exception. This makes them trivially testable in isolation.

File Functions Purpose
jwt.py create_access_token_for_user, get_username_from_token HS256 JWT encode/decode; 7-day TTL; all JWT errors normalized to ValueError
security.py generate_salt, get_password_hash, verify_password bcrypt via passlib
articles.py get_slug_for_article, check_article_exists, check_user_can_modify_article Slug generation, existence check, ownership predicate
authentication.py check_username_is_taken, check_email_is_taken Uniqueness guards; raise HTTP 400 on conflict
comments.py check_user_can_modify_comment Comment ownership predicate

Example — slug generation: A title like "Hello World!" becomes the slug "hello-world" via python-slugify. If that slug already exists in the database, check_article_exists raises HTTP 400 before any write occurs.

3.4 Repository / Data Access Layer

Location: app/db/repositories/ and app/db/queries/

Repositories are the only layer that talks to the database. Each repository wraps a single asyncpg.Connection (injected via DI) and exposes async methods for domain operations.

app/db/
├── repositories/
│   ├── base.py        ← BaseRepository: holds the connection, exposes self._conn
│   ├── users.py       ← UsersRepository: get_by_email, get_by_username, create, update
│   ├── profiles.py    ← ProfilesRepository: get_profile, follow/unfollow (transactional)
│   ├── articles.py    ← ArticlesRepository: CRUD, filter, feed, favorites
│   ├── comments.py    ← CommentsRepository: create, list, delete
│   └── tags.py        ← TagsRepository: get all tags
└── queries/
    ├── queries.py     ← aiosql loader — maps .sql files → Python callables
    ├── tables.py      ← PyPika table descriptors for dynamic query building
    └── sql/
        ├── users.sql, profiles.sql, articles.sql,
        ├── comments.sql, tags.sql
        └── ...        ← Named queries with aiosql sigils

aiosql is a library that loads .sql files and exposes named queries as Python async functions. A query named -- name: get-user-by-email^ in a .sql file becomes callable as queries.get_user_by_email(conn, email=...). The ^ sigil means "return one row"; * means "return all rows"; ! means "execute (INSERT/UPDATE/DELETE)"; <! means "execute and return the affected row".

PyPika is used for the one case where SQL cannot be statically written: filter_articles. The combination of optional filters (tag, author, favorited) produces different WHERE clauses, so PyPika builds the query object programmatically.

_get_article_from_db_record — domain assembly (and its N+1 tradeoff):

After fetching an article row, the repository performs additional queries to assemble the full domain object: - Fetch the author's Profile (including following status relative to requesting user) - Fetch favoritesCount and the favorited boolean for the requesting user

This is an intentional N+1 pattern — for each article row, 2 additional queries are issued. At the scale this reference implementation targets, this is acceptable, but at higher scale you would replace this with a single JOIN or a batch loader.

3.6 Configuration & Lifecycle

Location: app/core/

app/core/
├── config.py     ← get_app_settings() — @lru_cache factory
├── settings/
│   ├── app.py    ← AppSettings (Pydantic BaseSettings)
│   ├── dev.py    ← DevAppSettings (debug=True, generous CORS)
│   ├── prod.py   ← ProdAppSettings
│   └── test.py   ← TestAppSettings (test DB URL)
├── events.py     ← create_start_app_handler / create_stop_app_handler
└── logging.py    ← InterceptHandler (stdlib → Loguru bridge)

Environment selection: The APP_ENV environment variable (dev/prod/test) selects the settings class. get_app_settings() is decorated with @lru_cache, so settings are constructed once and reused.

Lifecycle hooks: FastAPI exposes on_startup and on_shutdown event hooks. Conduit uses these to: - Startup: Create the asyncpg connection pool (min_size=10, max_size=10) and attach it to app.state.pool - Shutdown: Close all connections in the pool gracefully

This means the pool is created once at process start and shared across all requests — individual connections are checked out per request via get_repository.

Logging: Loguru is used for structured, leveled logging. Since many libraries (uvicorn, asyncpg) emit to stdlib logging, InterceptHandler is installed to intercept all stdlib log records and forward them into Loguru's pipeline.

3.7 Integration Test Module (postman)

Location: postman/

postman/
├── run-api-tests.sh                   ← Shell entry point
└── Conduit.postman_collection.json    ← Declarative test collection

The test module is a black-box client. It has zero knowledge of the application's source code; it only knows the HTTP contract defined by the RealWorld spec.

How it works: 1. run-api-tests.sh reads APIURL from the environment to determine the target server 2. It generates a timestamp-based username/email (e.g., user1713350400@test.com) to avoid test identity collisions across runs 3. It invokes npx newman run Conduit.postman_collection.json with these values injected 4. Newman executes the collection sequentially: register → login → profile → article CRUD → favorites → follow/unfollow → comments 5. Each step captures response values (tokens, slugs, comment IDs) into collection-level variables for subsequent requests 6. Newman prints a pass/fail summary and exits non-zero on any failure

Important: Tests create real persistent state in the target database. In CI, APIURL should point at an ephemeral local instance to avoid polluting shared environments.

4. Data Flows

4.1 Request Lifecycle

Every HTTP request through the system follows this path:

sequenceDiagram
    participant Client
    participant Uvicorn
    participant Middleware
    participant DI as DI Layer (dependencies/)
    participant Handler as Route Handler
    participant Service as Service Layer
    participant Repo as Repository
    participant DB as PostgreSQL

    Client->>Uvicorn: HTTP Request
    Uvicorn->>Middleware: CORS / TrustedHost checks
    Middleware->>DI: Resolve Depends()
    DI->>DB: JWT → get_user_by_username (if auth required)
    DB-->>DI: User row
    DI->>DB: slug/id → domain object (if path param)
    DB-->>DI: Domain object row(s)
    DI-->>Handler: Injected User, Article, Repository, etc.
    Handler->>Service: Validate / authorize (pure logic)
    Service-->>Handler: OK or raises HTTPException
    Handler->>Repo: Async DB operation
    Repo->>DB: asyncpg + aiosql / PyPika SQL
    DB-->>Repo: Result rows
    Repo-->>Handler: Domain model(s)
    Handler-->>Client: Pydantic schema → JSON response

Error handling: Any HTTPException raised anywhere in the DI chain or handler is caught by http_error_handler and normalized to the RealWorld error envelope:

{ "errors": { "body": ["error message"] } }

Pydantic ValidationErrors (malformed request bodies) are caught by http422_error_handler and similarly normalized.

4.2 User Registration

sequenceDiagram

    participant Register as POST /api/users

    participant UsersRepo as UsersRepository
    participant JWTService as services/jwt.py

    Client->>Register: {username, email, password}
    Register->>AuthService: check_username_is_taken()

    Register->>UsersRepo: create_user_in_db(username, email, password)
    Note over UsersRepo: generate_salt() + get_password_hash()

Passwords are never stored in plaintext. The repository generates a random bcrypt salt, hashes the password, and stores only the salt and hash. The UserInDB domain model exposes a check_password(plain) method that re-derives the hash for comparison.

4.3 JWT Authentication

Every protected endpoint uses Depends(get_current_user_authorizer(required=True/False)):

flowchart TD
    A[HTTP Request arrives] --> B{Authorization header present?}
    B -- No + required=False --> C[return None\n anonymous context]
    B -- No + required=True --> D[raise HTTP 403]
    B -- Yes --> E[extract token from 'Bearer ']
    E --> F[PyJWT: decode HS256]
    F -- invalid/expired --> D
    F -- ok --> G[extract sub=username claim]
    G --> H[UsersRepository.get_user_by_username]
    H -- not found --> D
    H -- found --> I[return User domain object\ninjected into handler]

The token contains only {sub: username, exp: timestamp} — no roles or permissions are embedded. All authorization checks (ownership, etc.) are performed at request time against live DB data.

5. Infrastructure Dependencies

PostgreSQL

What it is: PostgreSQL is an open-source relational database. It provides ACID transactions, rich SQL query support, and reliable indexing.

The primary and only data store. All application state lives here — users, articles, comments, tags, and social graph relationships.

  • Driver: asyncpg — a high-performance async PostgreSQL driver for Python. It does not use an ORM; SQL is written explicitly.
  • Connection pool: Managed by the FastAPI lifecycle hooks. Pool size is min=10, max=10, configured in AppSettings. Each HTTP request checks out one connection from this pool.
  • Migrations: Managed by Alembic (see below). At runtime, the application only uses raw SQL via asyncpg — SQLAlchemy is present only for Alembic's DDL tooling.

Config:

Image: postgres:11.5-alpine
Port: 5432
DATABASE_URL: postgresql://postgres:postgres@db/postgres (Docker)
              postgresql://postgres:postgres@localhost/postgres (.env.example)
Data volume: ./postgres-data:/var/lib/postgresql/data

FastAPI / Uvicorn

What they are: FastAPI is a Python web framework that uses type hints and Pydantic for automatic request validation and OpenAPI schema generation. Uvicorn is an ASGI (Asynchronous Server Gateway Interface) server — the process that actually accepts TCP connections and passes them to FastAPI.

FastAPI handles routing, dependency injection, middleware, and response serialization. Uvicorn is the underlying process server (analogous to Gunicorn for WSGI, but async-native).

  • Port: 8000
  • API prefix: /api
  • Docs: /docs (Swagger UI), /redoc, /openapi.json
  • Middleware: CORS (all origins allowed), TrustedHost

Alembic

What it is: Alembic is a schema migration tool for Python projects. It tracks incremental changes to your database schema (tables, columns, indexes) in versioned migration scripts, similar to Rails migrations or Flyway.

Alembic runs at container startup before Uvicorn starts:

poetry run alembic upgrade head && poetry run uvicorn --host=0.0.0.0 app.main:app

This means every deployment automatically applies any pending schema migrations. Migration scripts live in app/db/migrations/. SQLAlchemy is used only for Alembic's DDL features — it is not used at runtime.

asyncpg + aiosql

asyncpg: Async PostgreSQL client library for Python. Significantly faster than psycopg2 due to its binary protocol implementation. aiosql: A small library that loads .sql files and exposes each named query as a Python async callable — keeping SQL out of Python string literals.

Together, these enable a "SQL-first" approach: all queries are written as plain SQL in app/db/queries/sql/*.sql files, making them easy to read, diff, and optimize, while still being callable as normal Python functions from repository methods.

aiosql query sigils: | Sigil | Meaning | |---|---| | ^ | Return one row | | * | Return all rows | | ! | Execute (INSERT/UPDATE/DELETE), return affected row count | | <! | Execute and return the modified row |

PyJWT + passlib/bcrypt

PyJWT: Encodes and decodes JSON Web Tokens — compact, self-contained tokens used for stateless authentication. passlib/bcrypt: A password hashing library. bcrypt is the algorithm; passlib is the Python wrapper providing a consistent API.

JWTs are HS256 (HMAC-SHA256) signed with a secret key from AppSettings. Tokens contain only {sub: username, exp: now+7d} — they are stateless and not stored server-side. Password hashes are generated with a per-user random salt and stored in the users table.

Pydantic

What it is: A Python data validation library that uses type annotations to define schemas and validate data at runtime. FastAPI is built on top of Pydantic.

Used for three distinct purposes: 1. Request/response schemas — validate incoming JSON bodies, serialize outgoing responses 2. Domain models — User, Article, Profile, Comment are Pydantic models 3. Settings management — AppSettings extends Pydantic's BaseSettings, which reads values from environment variables automatically

Docker / Docker Compose

What it is: Docker packages applications into portable container images. Docker Compose orchestrates multiple containers as a single application stack.

Two services are defined:

Service Image Role
app Built from Dockerfile (python:3.9.10-slim) FastAPI + Uvicorn
db postgres:11.5-alpine PostgreSQL

The app service depends on db and restarts on failure. The startup command runs migrations then starts the server.

RabbitMQ (via pika)

What it is: RabbitMQ is a message broker for asynchronous communication between services. The pika library is the Python AMQP client. Message brokers decouple producers from consumers — a service publishes a message to a queue without needing to know what will consume it.

pika is present in pyproject.toml dependencies, indicating intent to use RabbitMQ. However, no publish or consume call sites are visible in the current source files. Connection configuration (host, credentials) would typically come from environment variables. This dependency may be intended for future event-driven features (e.g., async notifications on article publish).

Newman (npx)

What it is: Newman is Postman's CLI test runner. It executes a Postman collection (a JSON file describing HTTP requests and assertions) from the command line without requiring the Postman GUI.

Newman is the sole runtime dependency of the postman module. It is invoked via npx (the Node.js package runner), requiring only Node.js to be installed — no explicit npm install step is needed.

7. Key Design Decisions

1. SQL-First Data Access (aiosql + asyncpg, not ORM)

All database queries are written as named plain SQL in .sql files rather than using an ORM like SQLAlchemy at runtime. This is a deliberate tradeoff:

  • Pro: Queries are readable, diffable, and optimizable by anyone who knows SQL. No ORM abstraction leaking into the data model.
  • Pro: asyncpg is significantly faster than ORM-based drivers due to binary protocol and connection pooling efficiency.
  • Con: More boilerplate for domain object assembly (see _get_article_from_db_record). No lazy loading — you must explicitly issue every query.
  • Exception: PyPika is used for the dynamic filter_articles query where the WHERE clause varies based on which query parameters are active. This avoids string concatenation while keeping the SQL generation safe and auditable.

2. Domain Models as Read-Side View Objects

Domain models (Article, Profile, Comment) are assembled by repositories and include viewer-relative computed fields (favorited, following). They are not ORM entities with lazy-loaded relationships — they are plain Pydantic models built from query results.

This means no "detached entity" bugs, no N+1 surprises hidden inside model property access, and no ORM session management. The tradeoff is that the repository layer bears responsibility for correctly assembling these objects.

3. Authorization at Two Layers

Ownership authorization is enforced redundantly: - DI layer: check_article_modification_permissions raises HTTP 403 if the requesting user is not the article author — this runs before the handler. - SQL layer: Some UPDATE/DELETE queries include WHERE author_id = :user_id in the WHERE clause — a defense-in-depth measure at the database level.

This dual enforcement means a misconfigured DI setup won't silently allow unauthorized writes through.

4. Stateless JWT Authentication

Tokens are not stored server-side. Every authenticated request decodes the JWT, extracts the username, and queries the database to hydrate the User object. This means: - No session store or Redis needed - Tokens cannot be individually revoked before expiry (7-day TTL) - Scaling horizontally is trivial — no shared session state between instances

5. Environment-Driven Configuration with Pydantic BaseSettings

AppSettings uses Pydantic's BaseSettings, which automatically reads from environment variables. The APP_ENV variable selects the settings subclass (dev/prod/test). @lru_cache on the factory ensures settings are parsed once per process. This means: - No configuration files to manage per environment — all config flows through environment variables - Settings are validated at startup (Pydantic will raise if required fields are missing or type-invalid) - Test settings can point at a separate database without code changes

6. Black-Box Integration Tests Only

The postman test module has zero visibility into application internals. It tests the HTTP contract only, which means: - Tests validate the actual user-facing behavior (not implementation details) - The test suite is portable — it can run against any deployed environment by changing APIURL - Tests are stateful and not idempotent — each run creates real database records; timestamp-based usernames reduce (but don't eliminate) collision risk

7. Alembic Migrations at Startup

Running alembic upgrade head as the first step in the container startup command ensures the database schema is always current before any requests are served. This is appropriate for a single-instance or rolling-deploy setup. For zero-downtime deployments with multiple instances starting simultaneously, this could cause a migration race — a future consideration for production hardening.

PostgreSQL

What it is PostgreSQL is an open-source relational database management system (RDBMS). It stores data in structured tables with rows and columns, enforces relationships between data (foreign keys), and supports advanced features like transactions (all-or-nothing operations), complex SQL queries, and indexing for fast lookups. If you are familiar with MySQL or Microsoft SQL Server, PostgreSQL is a direct counterpart.

How it is used here PostgreSQL is the single primary data store for the entire application. Every piece of persistent data — users, articles, comments, tags, and profiles — lives here. The application communicates with PostgreSQL through five repository classes:

  • ArticlesRepository
  • UsersRepository
  • ProfilesRepository
  • CommentsRepository
  • TagsRepository

Each repository issues SQL queries via the asyncpg driver (an async-native Python library for PostgreSQL). The connection pool — which keeps a set of pre-opened database connections ready to use, avoiding the overhead of opening a new connection on every request — is initialized in app/core/events.py and app/db/events.py, and made available to route handlers via FastAPI's dependency injection in app/api/dependencies/database.py.

FastAPI

What it is FastAPI is a modern Python web framework for building HTTP APIs. Its key differentiators are: it is built on Python's async/await model (so it can handle many requests concurrently without threads), it uses Python type hints to automatically validate request and response data, and it auto-generates interactive API documentation (via the OpenAPI standard) without any extra work.

How it is used here FastAPI is the backbone of the entire application. Every HTTP route the application exposes is defined using FastAPI's APIRouter in the app/api/routes/ directory, covering:

  • Authentication
  • Users
  • Profiles
  • Articles
  • Comments
  • Tags

The central app object is created in app/main.py. FastAPI's dependency injection system is used heavily — for example, app/api/dependencies/database.py provides a database connection to any route handler that requests it, without the handler needing to manage the connection lifecycle itself.

Uvicorn

What it is Uvicorn is the process that actually listens for incoming HTTP connections and hands them off to the FastAPI application. It implements the ASGI (Asynchronous Server Gateway Interface) standard — the async equivalent of the older WSGI standard used by frameworks like Django and Flask. Think of Uvicorn as the "engine" and FastAPI as the "application logic" it runs. Without Uvicorn (or an equivalent), FastAPI has no way to receive HTTP traffic.

How it is used here Uvicorn is launched as the final command when the Docker container starts, after database migrations have been applied:

poetry run uvicorn --host=0.0.0.0 app.main:app

This tells Uvicorn to import the app object from app/main.py and serve it. Setting host=0.0.0.0 ensures the server accepts connections from outside the container (not just from within it). Uvicorn's access and ASGI loggers are also monitored by the application's logging configuration.

Alembic

What it is Alembic is a database schema migration tool. As an application evolves, its database schema changes — tables are added, columns are renamed, indexes are created, and so on. Alembic tracks these changes as versioned migration scripts, so the database schema can be evolved incrementally and reproducibly across all environments (local development, staging, production). It is the standard migration tool for Python projects that use SQLAlchemy.

How it is used here Migration scripts live in app/db/migrations/. Every time the Docker container starts, Alembic runs before the web server starts:

poetry run alembic upgrade head

upgrade head means "apply every migration that has not yet been applied, up to the latest version." This guarantees the database schema is always current when the application starts handling requests, with no manual migration step required after deployment.

RabbitMQ (via pika)

What it is RabbitMQ is a message broker — a middleman service that allows different parts of a system to communicate asynchronously by sending and receiving messages via queues. Instead of Service A calling Service B directly and waiting for a response, Service A puts a message on a queue and continues. Service B picks up the message and processes it independently. This decouples services and improves resilience. The pika library is the Python client used to talk to RabbitMQ using the AMQP protocol (Advanced Message Queuing Protocol).

How it is used here pika is included as a declared dependency in pyproject.toml/poetry.lock, confirming that asynchronous messaging via RabbitMQ is part of this project's design. The specific locations in source code where messages are published or consumed are not fully visible in the provided files, and connection configuration (hostname, credentials, virtual host) is expected to be supplied via environment variables at runtime.

For the next engineer: If you are extending this system, look for pika.BlockingConnection, pika.SelectConnection, or similar in the codebase to find where messages are sent and received. Connection parameters are likely loaded from environment variables — check .env.example or the application settings for RABBITMQ_URL or similar keys.

Docker

What it is Docker packages an application and all of its dependencies (language runtime, libraries, system tools) into a self-contained unit called a container image. This image can be run identically on any machine with Docker installed, eliminating environment-specific bugs ("it works on my machine"). docker-compose is a companion tool that defines and runs multiple containers together as a single application stack using a docker-compose.yml file.

How it is used here Two files define the containerized environment:

  • Dockerfile — Describes how to build the application image. It starts from a minimal Python 3.9 base image, installs system dependencies (netcat), installs the project's Python dependencies using Poetry (production only, no dev tools), copies the application source code, and defines the startup command.
  • docker-compose.yml — Defines two services that run together:
  • app — The FastAPI application container built from the Dockerfile.
  • db — A PostgreSQL container using the official postgres:11.5-alpine image.

The app service is configured with depends_on: db, meaning docker-compose will start the db container first. The app service restarts automatically if it crashes (restart: on-failure). PostgreSQL data is persisted to the host filesystem via a bind-mounted volume so data survives container restarts.
