What is this?

Conduit is a Medium-like blogging platform backend — a JSON REST API that lets users register, write and publish articles, leave comments, follow other writers, and curate content through tags and favorites. It's a fully async Python service backed by PostgreSQL.

The codebase implements the open RealWorld/Conduit spec, a community standard for building realistic "not just a todo list" demo apps. That means the API contract is well-defined, externally documented, and validated by an included integration test suite.

Why does it exist?

Conduit gives teams a production-shaped starting point for a content platform. It solves the "blank page" problem: auth, articles, social graphs, and tagging are all wired up from day one, so you can focus on extending the product rather than plumbing basics.

Concrete example: Maya is a developer building a technical newsletter platform. She forks Conduit, runs docker-compose up, and within minutes has a working API that handles user sign-up, article authorship, and reader follows. She points her frontend at http://localhost:8000/api, confirms the Postman integration tests pass, then starts layering in her newsletter-specific features — rather than spending the first two weeks writing auth middleware and tag systems from scratch.

A few things worth knowing before you dig in:

  • Viewer-relative fields. Fields like favorited and following are computed per request based on who's calling. Every Article and Profile is context-aware.
  • No ORM at runtime. Queries use raw asyncpg with SQL files (aiosql) and a dynamic query builder (PyPika) for filtered lists. SQLAlchemy only appears in Alembic migrations.
  • N+1 heads-up. ArticlesRepository fires 3–4 extra queries per article to hydrate tags, favorites, and author profiles. Fine for now; worth knowing before you hit list endpoints at scale.
  • Fresh JWT on every user response. GET /user and PUT /user re-issue a new token so claims stay current.
  • Integration tests are external. postman/run-api-tests.sh spins up Newman as an HTTP client against a live API instance — no internal coupling.

1. System Overview

Conduit is a Medium-like blogging platform backend. It exposes a JSON REST API over HTTP, backed by PostgreSQL, and is packaged as a Docker container. A separate Newman-based test harness validates the full API contract from the outside.

The two top-level modules are entirely decoupled at the code level — postman communicates with app exclusively over HTTP.

2. Component Descriptions

2.1 FastAPI Application (app)

The core backend. It is structured as a four-layer application: routing, dependencies/guards, services, and data access. Each layer has a clear, single responsibility.

Layer Location Responsibility
HTTP / Routing app/api/routes/ Parse requests, serialize responses, enforce camelCase JSON envelopes
Dependencies & Guards app/api/dependencies/ JWT authentication, path-parameter resolution, ownership authorization
Services app/services/ Pure business logic: slug generation, uniqueness checks, password hashing, JWT minting
Data Access app/db/repositories/ Repository-pattern classes that execute SQL against PostgreSQL via asyncpg

Domain areas covered: - User registration, login, and profile management - Article authoring (CRUD), tagging, and favoriting - Comment creation and deletion - Social following graph - Tag listing

Key characteristic: All domain objects (Article, Profile, Comment) are viewer-relative — fields like favorited and following are resolved at query time based on the authenticated caller. The same article row returns favorited: true for one user and favorited: false for another.

2.2 Integration Test Suite (postman)

A Newman-based CLI harness that acts as a black-box HTTP client running the full Conduit API contract. It has no code-level coupling to app.

File Role
run-api-tests.sh Entry point; configures Newman and fires it
Conduit.postman_collection.json The full test contract (request definitions + assertions)

Key characteristics: - APIURL, USERNAME, EMAIL, and PASSWORD are all injectable via environment variables — the same script works locally and in CI pipelines. - Credentials are timestamped by default (user_<epoch>) to prevent collisions when run against shared environments. - A 500 ms inter-request delay (--delay-request 500) prevents race conditions and handles any API throttling.

4.3 Filtered Article Listing (N+1 Pattern)

This flow highlights the most significant performance characteristic of the codebase. For each article returned by the main query, three or more additional queries are fired to hydrate viewer-relative fields.

sequenceDiagram
    actor Client
    participant RT as Route: GET /articles
    participant ART as ArticlesRepository
    participant DB as PostgreSQL

    Client->>RT: GET /articles?tag=python&author=alice&limit=20&offset=0
(Bearer token optional)

    RT->>ART: filter_articles(tag, author, favorited, limit, offset, current_user)
    Note over ART: PyPika builds dynamic SQL:
JOINs and WHERE clauses
added conditionally per filter

    ART->>DB: SELECT articles ... (1 paginated query)
    DB-->>ART: raw article records [r1, r2, ..., rN]

    loop For each article record rᵢ
        ART->>DB: SELECT tags WHERE article_id = rᵢ.id
        ART->>DB: SELECT COUNT(*) FROM article_favorites WHERE article_id = rᵢ.id
        ART->>DB: SELECT is_favorited WHERE article_id=? AND user_id=?  (if authenticated)
        ART->>DB: SELECT profile + following WHERE user_id = rᵢ.author_id
        DB-->>ART: hydration data for rᵢ
    end

    ART-->>RT: List[Article] (viewer-relative fields populated)
    RT-->>Client: 200 {"articles": [...], "articlesCount": N}

⚠️ Performance Note: With 20 articles per page, this results in up to 81 database round-trips for a single GET /articles request (1 + 20×4). See Known Issues for context.

6. Infrastructure Dependencies

PostgreSQL

What it is: A battle-tested open-source relational database. Stores structured data in tables with strong ACID guarantees.

How it's used here: - Primary data store for all application state: users, articles, comments, tags, followers, favorites. - Accessed at runtime via asyncpg (a pure-async PostgreSQL driver — not SQLAlchemy) for maximum async throughput. - Schema migrations are managed separately by Alembic (see below).

Configuration:

Image: postgres:11.5-alpine
Port:  5432:5432
URL:   postgresql://postgres:postgres@db/postgres
Pool:  min=10, max=10 connections
Data:  persisted to ./postgres-data volume

asyncpg

What it is: A high-performance, fully async PostgreSQL client library for Python. Unlike psycopg2, it speaks the PostgreSQL binary protocol directly and integrates natively with Python's asyncio.

How it's used here: - The connection pool (asyncpg.Pool) is created at application startup and stored on app.state.pool. - The get_repository() DI factory acquires a connection from the pool per-request and injects it into the relevant BaseRepository subclass. - All SQL queries are executed asynchronously — no blocking I/O on any database call.

FastAPI + Starlette

What it is: FastAPI is a modern Python web framework built on top of Starlette (ASGI toolkit). It provides automatic OpenAPI docs, Pydantic-native request validation, and a powerful dependency injection system.

How it's used here: - All HTTP routing is defined with FastAPI decorators (@router.get, @router.post, etc.). - The Depends() system is used extensively for authentication, object resolution, and permission guards — keeping route handlers thin. - Middleware (CORS, TrustedHost) is registered in app/main.py.

Uvicorn

What it is: An ASGI server — the runtime that accepts HTTP connections and calls into the FastAPI application. Think of it as the equivalent of gunicorn for ASGI apps.

How it's used here: - Runs as the production server: uvicorn app.main:app --host=0.0.0.0. - Configured to listen on port 8000, exposed by Docker.

Pydantic

What it is: A Python data validation library using type annotations. In newer versions it's built on Rust for performance.

How it's used here: - Settings management — BaseAppSettings subclasses read from .env files and environment variables. - Request/response schemas — RWSchema subclasses validate incoming JSON and serialize outgoing responses with camelCase field aliasing. - Domain models — RWModel subclasses define the internal domain objects (User, Article, etc.).

PyJWT + passlib/bcrypt

What they are: PyJWT encodes/decodes JSON Web Tokens. passlib is a password hashing library; bcrypt is a deliberately slow hashing algorithm designed to resist brute-force attacks.

How they're used here: - jwt.py mints HS256 JWTs containing the username, with a 7-day TTL. A fresh token is issued on every /user GET or PUT response. - security.py uses passlib's CryptContext(schemes=["bcrypt"]) for all password hashing and verification. bcrypt generates and stores a salt internally (the "salt" column in the users table is a legacy field; bcrypt hashes already encode the salt).

aiosql + PyPika

What they are: - aiosql is a library that loads .sql files and makes named queries available as Python function calls — keeping SQL in .sql files rather than embedded strings. - PyPika is a Python SQL query builder that constructs SQL programmatically, avoiding string concatenation. Think of it as a type-safe str.format() for SQL.

How they're used here: - Static, well-known queries (user lookups, comment inserts, etc.) live in app/db/queries/sql/*.sql and are loaded by aiosql. - Dynamic queries — specifically article filtering where the WHERE clause varies by tag, author, and favorited query parameters — are built at runtime by PyPika in ArticlesRepository.

Alembic + SQLAlchemy

What they are: Alembic is a database migration tool. SQLAlchemy is a Python ORM/SQL toolkit; Alembic depends on it for connection management in its migration environment.

How they're used here: - Only for migrations — SQLAlchemy does not appear in any runtime request path. - Migrations live in app/db/migrations/ and are applied automatically at container startup (alembic upgrade head). - alembic.ini configures the migrations directory and SQL logging.

Newman (npx newman)

What it is: Newman is the CLI runner for Postman collections. It executes a .json collection file as a test suite against a live HTTP target.

How it's used here: - The postman/run-api-tests.sh script invokes npx newman run with the Conduit collection, injecting the target URL and test credentials as Newman global variables.

RabbitMQ (pika)

What it is: RabbitMQ is a message broker implementing the AMQP protocol. It enables asynchronous, decoupled communication between services via named queues. pika is the Python AMQP client library.

8. Key Design Decisions

1. No ORM at Runtime

SQLAlchemy is intentionally excluded from the runtime request path. Instead, the stack uses:

  • aiosql for static, file-based named queries
  • PyPika for dynamic query construction (article filtering)
  • asyncpg for raw async execution

Why: Avoids ORM abstraction overhead and keeps database behavior explicit and auditable. The trade-off is more verbose repository code and no automatic lazy loading (which also avoids hidden query surprises — except for the intentional N+1 described below).

Implication for you: When tracing a bug to a SQL query, look in app/db/queries/sql/*.sql for static queries and app/db/repositories/articles.py for dynamically built ones.

2. Viewer-Relative Domain Objects

Every Article, Profile, and Comment object carries context-sensitive boolean fields (favorited, following) that are hydrated per-request based on the authenticated caller.

# Example: same article, different callers
# Alice has favorited this article and follows the author:
{"slug": "hello-world", "favorited": true, "author": {"following": true}}

# Anonymous visitor:
{"slug": "hello-world", "favorited": false, "author": {"following": false}}

Why: Aligns with the RealWorld spec and avoids client-side joins. The trade-off is the N+1 query pattern (see below) and that domain objects cannot be cached without caller context.

3. Deliberate N+1 Hydration in ArticlesRepository

Rather than writing a single large JOIN query, _get_article_from_db_record() fires 3–4 targeted queries per article (tags, favorites count, is-favorited, author profile).

Why this might be acceptable: - Queries are simple and individually fast. - The connection pool (size 10) handles concurrency. - The codebase prioritizes clarity over micro-optimization.

Why to monitor it: At limit=20, a single GET /articles call can generate ~81 queries. Under load, this will exhaust the connection pool quickly. See Known Issues.

4. Fresh JWT on Every User Response

Every GET /user and PUT /user response re-issues a new JWT token with a refreshed 7-day TTL.

Why: Keeps token claims (e.g., email, username) current if a user updates their profile. Clients should always store the most recently returned token.

Implication: There is no token revocation mechanism. A stolen token is valid for up to 7 days.

5. Dependency Injection for Auth, Object Resolution, and Permissions

FastAPI's Depends() system is used as a three-tier gate before any route handler executes:

flowchart LR
    A["Incoming Request"] --> B["1. Auth\nget_current_user_authorizer(required=True/False)"]
    B --> C["2. Object Resolution\nget_article_by_slug_from_path\nget_comment_by_id_from_path"]
    C --> D["3. Permission Guard\ncheck_article_modification_permissions"]
    D --> E["Route Handler\n(thin, mostly schema mapping)"]

This keeps route handlers free of auth/authorization logic and makes the security model declarative and composable.

6. Settings Selected by APP_ENV at Process Start

The get_app_settings() factory reads APP_ENV once and returns the appropriate AppSettings subclass, cached for the process lifetime. This means: - No settings hot-reload. - Changing APP_ENV requires a process restart. - Test environments (APP_ENV=test) can point at a separate database without code changes.

Base paths: All /feed, /{slug}/favorite, and article resource endpoints are mounted under /articles. Authentication endpoints are mounted under /users.

Authentication: Protected endpoints require a JWT Bearer token in the Authorization header: Authorization: Token <jwt>. Tokens are HS256-signed, have a 7-day TTL, and are obtained via POST /users or POST /users/login.

Serialization: All JSON responses use camelCase field names, enforced via RWSchema Pydantic aliases.

GET /articles/feed

Description

Returns a paginated list of articles authored by users that the currently authenticated user follows. This is the personalized social feed — think "timeline" — and is restricted to authenticated callers only. Articles are ordered by created_at descending (newest first).

Because each article record requires supplementary data (tags, favorites count, author profile), the underlying implementation fires N+1 queries per article record after the initial feed query. The following flag on every returned article's author object is always true by construction: all feed authors are followed by the caller.

Authentication: Required — Authorization: Token <jwt>

Query Parameters

Parameter Type Required Default Description
limit integer No 20 Maximum number of articles to return. Use for paging.
offset integer No 0 Number of articles to skip. Use with limit to page through results.

Request Body: None

Response — 200 OK

{
  "articles": [
    {
      "slug": "how-to-train-your-dragon",
      "title": "How to train your dragon",
      "description": "Ever wonder how?",
      "body": "It takes a Herculean effort.",
      "tagList": ["dragons", "training"],
      "createdAt": "2026-04-10T12:00:00.000Z",
      "updatedAt": "2026-04-10T12:00:00.000Z",
      "favorited": false,
      "favoritesCount": 3,
      "author": {
        "username": "dragonmaster",
        "bio": "I ride dragons for fun.",
        "image": "https://example.com/avatar.jpg",
        "following": true
      }
    }
  ],
  "articlesCount": 1
}
Field Type Description
articles array Ordered list of fully-hydrated article objects.
articles[].slug string URL-safe unique identifier for the article.
articles[].title string Article title.
articles[].description string Short summary / subtitle.
articles[].body string Full Markdown content of the article.
articles[].tagList array List of tag strings associated with the article.
articles[].createdAt string ISO 8601 UTC timestamp of creation.
articles[].updatedAt string ISO 8601 UTC timestamp of last update.
articles[].favorited boolean Whether the calling user has favorited this article. Always viewer-relative.
articles[].favoritesCount integer Total number of users who have favorited this article.
articles[].author object Profile of the article's author.
articles[].author.following boolean Always true in feed responses (feed is scoped to followed authors).
articlesCount integer Total matching articles (used for pagination UI).

Error Responses

Status Condition
401 Authorization header is missing or the JWT is invalid/expired.

Data Flow

  1. Client sends GET /articles/feed with a valid Bearer token.
  2. RWAPIKeyHeader extracts the token; jwt.get_username_from_token() decodes the username; UsersRepository.get_user_by_username() fetches the caller's UserInDB. Returns HTTP 401 on failure.
  3. DI resolves ArticlesRepository from the asyncpg connection pool.
  4. ArticlesRepository.get_articles_for_user_feed() executes: sql SELECT articles.* FROM articles WHERE articles.author_id IN ( SELECT following_id FROM followers WHERE follower_id = :current_user_id ) ORDER BY articles.created_at DESC LIMIT :limit OFFSET :offset;
  5. For each returned row, _get_article_from_db_record() fires N+1 hydration queries:
  6. SELECT tags WHERE article_id = ?
  7. SELECT COUNT(*) FROM article_favorites WHERE article_id = ?
  8. SELECT is_favorited FROM article_favorites WHERE article_id = ? AND user_id = ?
  9. SELECT profile WHERE user_id = author_id (with following = true for all feed authors)
  10. ArticleListResponse Pydantic schema serializes the hydrated list.
  11. FastAPI returns HTTP 200 with camelCase JSON.

Example Request

curl -X GET "https://api.example.com/articles/feed?limit=5&offset=0" \
     -H "Authorization: Token eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."

POST /articles/{slug}/favorite

Description

Marks an article as a favorite for the currently authenticated user. This inserts a row into the article_favorites junction table associating the caller's user_id with the target article's article_id. The response returns the fully re-hydrated article reflecting the updated favoritesCount and favorited: true.

Calling this endpoint on an article the user has already favorited is a no-op at the route level (behavior depends on the INSERT INTO article_favorites constraint — duplicate inserts are silently ignored or raise an integrity error depending on database constraints).

GET /articles

Description

Returns a paginated, filtered list of all articles in the system. Authentication is optional: unauthenticated callers receive articles with favorited: false and author.following: false for all entries. Authenticated callers receive viewer-relative favorited and author.following values computed against their identity.

Three mutually combinable filter dimensions are supported via query parameters. Filters are applied as AND conditions. Internally, ArticlesRepository.filter_articles() uses PyPika to construct the query dynamically, adding JOIN clauses and WHERE predicates only for supplied filter parameters.

⚠️ N+1 Warning: For each article returned by the primary query, four additional SQL queries are fired for hydration (tags, favorites count, is_favorited, author profile). For large limit values this can result in significant query volume. Prefer smaller page sizes in production.

Authentication: Optional — Authorization: Token <jwt>

Query Parameters

Parameter Type Required Default Description
tag string No — Filter articles that have this tag. Matches exactly one tag string.
author string No — Filter articles written by this username.
favorited string No — Filter articles that have been favorited by this username.
limit integer No 20 Maximum number of articles to return.
offset integer No 0 Number of articles to skip. Use with limit for pagination.

Request Body: None

Response — 200 OK

{
  "articles": [
    {
      "slug": "how-to-train-your-dragon",
      "title": "How to train your dragon",
      "description": "Ever wonder how?",
      "body": "It takes a Herculean effort.",
      "tagList": ["dragons", "training"],
      "createdAt": "2026-04-10T12:00:00.000Z",
      "updatedAt": "2026-04-10T12:00:00.000Z",
      "favorited": false,
      "favoritesCount": 3,
      "author": {
        "username": "dragonmaster",
        "bio": "I ride dragons for fun.",
        "image": "https://example.com/avatar.jpg",
        "following": false
      }
    }
  ],
  "articlesCount": 42
}
Field Type Description
articles array Ordered list of hydrated article objects (see field schema in feed endpoint).
articlesCount integer Total number of articles matching the applied filters, ignoring limit/offset. Useful for building pagination controls.

Error Responses

Status Condition
401 A malformed token was supplied (valid absence is allowed).
422 Query parameter validation failed (e.g., non-integer limit).

Data Flow

  1. Client sends GET /articles with optional query parameters and an optional Bearer token.
  2. Optional JWT auth dependency: if a Bearer token is present, it is decoded to a username and UsersRepository.get_user_by_username() fetches UserInDB; otherwise current_user = None.
  3. DI resolves ArticlesRepository from the asyncpg pool.
  4. ArticlesRepository.filter_articles() constructs a dynamic PyPika query:
  5. If tag provided: JOIN article_tags at ON at.article_id = a.id JOIN tags t ON t.id = at.tag_id WHERE t.tag = :tag
  6. If author provided: JOIN users u ON u.id = a.author_id WHERE u.username = :author
  7. If favorited provided: JOIN article_favorites af ON af.article_id = a.id JOIN users u2 ON u2.id = af.user_id WHERE u2.username = :favorited
  8. LIMIT :limit OFFSET :offset
  9. For each article record, _get_article_from_db_record() fires N+1 hydration:
  10. SELECT tags WHERE article_id = ?
  11. SELECT COUNT(*) FROM article_favorites WHERE article_id = ?
  12. If authenticated: SELECT ... WHERE article_id = ? AND user_id = ? for favorited flag
  13. SELECT profile WHERE user_id = author_id with viewer-relative following
  14. ArticleListResponse serializes the result list.
  15. HTTP 200 returned.

Example Requests

# Unauthenticated — all recent articles
curl "https://api.example.com/articles?limit=10&offset=0"

# Filter by tag, authenticated
curl "https://api.example.com/articles?tag=dragons&limit=5" \
     -H "Authorization: Token eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."

# Articles favorited by a specific user
curl "https://api.example.com/articles?favorited=dragonmaster"

POST /articles

Description

Creates a new article authored by the authenticated user. The article's URL slug is automatically generated from the title using python-slugify. Tags supplied in tagList are upserted into the tags table and linked to the new article via the article_tags junction table.

The endpoint rejects creation if a slug derived from the given title conflicts with an existing article slug, returning HTTP 400. Slug generation is deterministic: the same title will always produce the same slug.

Authentication: Required — Authorization: Token <jwt>

Request Body

Content-Type: application/json

{
  "article": {
    "title": "How to train your dragon",
    "description": "Ever wonder how?",
    "body": "It takes a Herculean effort.",
    "tagList": ["dragons", "training"]
  }
}
Field Type Required Description
article.title string Yes Article title. Used to auto-generate the slug via python-slugify.
article.description string Yes Short summary or subtitle of the article.
article.body string Yes Full content of the article (Markdown supported).
article.tagList array No List of tag strings. Tags are created if they don't already exist (bulk UPSERT).

Response — 201 Created

{
  "article": {
    "slug": "how-to-train-your-dragon",
    "title": "How to train your dragon",
    "description": "Ever wonder how?",
    "body": "It takes a Herculean effort.",
    "tagList": ["dragons", "training"],
    "createdAt": "2026-04-17T09:00:00.000Z",
    "updatedAt": "2026-04-17T09:00:00.000Z",
    "favorited": false,
    "favoritesCount": 0,
    "author": {
      "username": "jane",
      "bio": "Writer and adventurer.",
      "image": null,
      "following": false
    }
  }
}

Error Responses

Status Condition
400 An article with the derived slug already exists.
401 Authorization token missing or invalid.
422 Request body failed Pydantic validation (e.g., missing required field).

Data Flow

  1. Client sends POST /articles with JSON body and a valid Bearer token.
  2. Required JWT auth dependency extracts and verifies the token; fetches UserInDB for the author. Returns HTTP 401 on failure.
  3. Pydantic validates the body against ArticleInCreate schema; returns HTTP 422 on invalid input.
  4. DI resolves ArticlesRepository and TagsRepository from the pool.
  5. articles.generate_slug_for_article(title) converts the title to a URL slug via python-slugify.
  6. articles.check_article_exists_for_slug(slug) queries the articles table; raises HTTP 400 if slug conflicts.
  7. TagsRepository.get_or_create_tags(tagList) performs: sql INSERT INTO tags (tag) VALUES (:tag) ON CONFLICT (tag) DO NOTHING RETURNING *;
  8. ArticlesRepository.create_article() executes: sql INSERT INTO articles (slug, title, description, body, author_id) VALUES (...) RETURNING *; INSERT INTO article_tags (article_id, tag_id) VALUES (...); -- one row per tag
  9. _get_article_from_db_record() hydrates: tags, favorites count (0), favorited = false, author profile.
  10. ArticleResponse serializes the hydrated object.
  11. FastAPI returns HTTP 201.

POST /users/login

Description

Authenticates an existing user by email and password, returning a fresh JWT access token. The password is verified by bcrypt-comparing the submitted plaintext against the stored hashed_password (which is salted per-user). No session state is stored server-side; all authentication state lives in the signed JWT.

The issued token is an HS256-signed JWT containing the user's username as the subject, with a 7-day TTL (exp claim). This same token is used as the Bearer token in the Authorization header for all protected endpoints.

Authentication: None (this is the login endpoint)

Request Body

Content-Type: application/json

{
  "user": {
    "email": "jane@example.com",
    "password": "supersecretpassword"
  }
}
Field Type Required Description
user.email string Yes The registered email address of the user.
user.password string Yes The user's plaintext password for bcrypt verification.

Response — 200 OK

{
  "user": {
    "username": "jane",
    "email": "jane@example.com",
    "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiJqYW5lIiwiZXhwIjoxNzQ2NTI4MDAwfQ.abc123",
    "bio": "Writer and adventurer.",
    "image": "https://example.com/jane.jpg"
  }
}
Field Type Description
user.username string The user's unique username.
user.email string The user's email address.
user.token string HS256 JWT. Include as Authorization: Token <token> on subsequent protected requests. 7-day TTL.
user.bio string/null The user's biography text, or null if not set.
user.image string/null URL to the user's avatar image, or null if not set.

Error Responses

Status Condition
400 No account found for the provided email, or the password is incorrect. The error message deliberately conflates the two cases to avoid user enumeration.
422 Request body failed Pydantic validation (e.g., malformed JSON, missing field).

Data Flow

  1. Client sends POST /users/login with JSON credentials.
  2. CORS and TrustedHost middleware pass the request through.
  3. Pydantic validates the body as UserLoginIn schema; returns HTTP 422 on invalid structure.
  4. DI resolves UsersRepository from the asyncpg pool.
  5. UsersRepository.get_user_by_email(email) executes: sql SELECT * FROM users WHERE email = :email; Raises HTTP 400 if no record is found.
  6. UserInDB.check_password(password) calls security.verify_password(plaintext, hashed_password) — bcrypt re-hashes the plaintext with the stored salt and compares digests. Raises HTTP 400 on mismatch.
  7. jwt.create_access_token(username) mints a fresh HS256 JWT: python {"sub": username, "exp": datetime.utcnow() + timedelta(days=7)}
  8. UserWithTokenResponse Pydantic schema wraps user fields and the token.
  9. FastAPI serializes camelCase JSON and returns HTTP 200. No database writes occur.

Example Request

curl -X POST "https://api.example.com/users/login" \
     -H "Content-Type: application/json" \
     -d '{
       "user": {
         "email": "jane@example.com",
         "password": "supersecretpassword"
       }
     }'

Using the token in a subsequent request:

Data Flow

  1. Client sends POST /users with JSON registration data.
  2. CORS and TrustedHost middleware pass the request through.
  3. Pydantic validates the body as UserRegistrationIn schema; returns HTTP 422 on invalid input.
  4. DI factory get_repository(UsersRepository) acquires an asyncpg connection from app.state.pool.
  5. authentication.check_email_is_taken(email) executes: sql SELECT 1 FROM users WHERE email = :email LIMIT 1; Raises HTTP 400 if a record is found.
  6. authentication.check_username_is_taken(username) executes: sql SELECT 1 FROM users WHERE username = :username LIMIT 1; Raises HTTP 400 if a record is found.
  7. UserInDB.change_password(password) calls security.get_password_hash(password):
  8. bcrypt generates a random salt.
  9. bcrypt produces hashed_password = bcrypt(salt + password).
  10. Both salt and hashed_password are stored.
  11. UsersRepository.create_user_in_db() executes inside an asyncpg transaction: sql INSERT INTO users (id, username, email, salt, hashed_password, bio, image) VALUES (:id, :username, :email, :salt, :hashed_password, NULL, NULL) RETURNING *;
  12. jwt.create_access_token(username) mints an HS256 JWT with a 7-day exp claim.
  13. UserWithTokenResponse schema wraps the new user object with the embedded token.
  14. FastAPI serializes camelCase JSON and returns HTTP 201.

Architecture Overview

This project is a containerized Python web API backed by a relational database. HTTP requests are handled by FastAPI (the web framework) served through Uvicorn (the application server). PostgreSQL stores all persistent data. Alembic manages schema changes automatically on startup. RabbitMQ is available as a message broker for asynchronous workloads. All services are orchestrated via Docker and Docker Compose, and Python dependencies are managed by Poetry.

Components

Docker

What it is Docker is a containerization platform that packages an application and all of its dependencies (runtime, libraries, config) into a single portable unit called a container image. This guarantees that the application behaves identically whether it's running on a developer's laptop, a CI server, or production infrastructure — there is no "works on my machine" problem.

How it's used here The project uses two Docker artifacts:

  • Dockerfile — Builds the application container image. It starts from a minimal Python base image, installs system tools and Poetry, installs Python dependencies, and defines the command that runs when the container starts.
  • docker-compose.yml — Defines and wires together the full local stack as two services: app (the FastAPI application) and db (PostgreSQL). Compose ensures db is healthy before app starts, manages port exposure, and injects environment variables from a .env file.

What it is FastAPI is a modern Python web framework for building HTTP APIs. Key characteristics that differentiate it from older frameworks like Flask or Django REST Framework:

  • Async-native — Built on Python's async/await model, which allows the server to handle many concurrent requests efficiently without blocking threads.
  • Automatic documentation — It reads Python type annotations and generates interactive API docs (Swagger UI at /docs, ReDoc at /redoc) with zero extra configuration.
  • Dependency injection — It has a built-in system for declaring reusable components (e.g., "this route requires an authenticated user") that are automatically resolved and injected into route handler functions.

How it's used here FastAPI is the core HTTP layer of the application. Every API endpoint — authentication, users, profiles, articles, comments, and tags — is implemented as a FastAPI route. The project is organized as follows:

  • app/api/routes/ — Contains individual route files grouped by resource (e.g., users, articles).
  • app/api/dependencies/ — Contains reusable dependency functions for database session access, authentication/authorization, and request validation. These are injected into routes using FastAPI's dependency injection system.
  • app/main.py — The application entrypoint that creates the FastAPI instance and registers all routers.

Uvicorn

What it is Uvicorn is an ASGI (Asynchronous Server Gateway Interface) server for Python. Think of it as the low-level network layer that sits in front of FastAPI: it listens for incoming TCP connections on a port, handles the HTTP protocol, and hands requests off to the FastAPI application. It is the Python equivalent of Gunicorn (for WSGI/synchronous frameworks) but built for async workloads.

ASGI vs WSGI: Traditional Python web servers use WSGI, which is synchronous. ASGI is the async successor, required for frameworks like FastAPI that use async/await.

How it's used here Uvicorn is the production server that actually serves the FastAPI application. It is launched as the final step in the container startup command in the Dockerfile:

poetry run uvicorn --host=0.0.0.0 app.main:app

This tells Uvicorn to load the app object from the app/main.py module and start listening for HTTP traffic on all network interfaces.

Poetry

What it is Poetry is a Python dependency management and packaging tool. It replaces the traditional combination of pip + requirements.txt + virtualenv with a single, unified tool that:

  • Declares dependencies and version constraints in pyproject.toml (human-readable and editable).
  • Generates a poetry.lock file (a precise, reproducible snapshot of every resolved package version).
  • Manages isolated virtual environments automatically.

How it's used here Poetry manages all Python library dependencies for the project. Inside the Docker build:

  1. Poetry itself is installed via pip install poetry==1.1.
  2. poetry install --no-dev installs only production dependencies (skipping development/testing tools) as declared in pyproject.toml and pinned in poetry.lock.
  3. All runtime commands (alembic, uvicorn) are executed via poetry run ..., which ensures they run inside the correct virtual environment.

PostgreSQL

What it is PostgreSQL (often called "Postgres") is an open-source relational database management system (RDBMS). It stores data in structured tables with rows and columns, enforces relationships and constraints between tables, and is queried using SQL. PostgreSQL is known for its standards compliance, reliability, support for complex queries, and ACID transactional guarantees (Atomicity, Consistency, Isolation, Durability).

How it's used here PostgreSQL is the sole persistent data store for the application. It holds all domain data:

Alembic

What it is Alembic is a database schema migration tool designed for use with SQLAlchemy (a popular Python ORM/database toolkit). As an application evolves, its database schema changes — new tables are added, columns are modified, indexes are created. Alembic tracks these changes as versioned migration scripts, allowing the schema to be evolved incrementally and deterministically across all environments without manual SQL intervention.

Analogy: Alembic is to your database schema what Git is to your source code — a version-controlled history of changes that can be applied, rolled back, and shared across the team.

How it's used here Migration scripts live in app/db/migrations/. Alembic is run automatically at container startup, before Uvicorn launches, via the Dockerfile command:

poetry run alembic upgrade head

upgrade head applies all pending migrations to bring the database schema up to the latest version. This ensures the database schema is always in sync with the application code whenever the container starts, without any manual operator intervention.

RabbitMQ

What it is RabbitMQ is an open-source message broker that implements the AMQP (Advanced Message Queuing Protocol). A message broker decouples services by acting as an intermediary: one service publishes a message to a queue without needing to know who will process it, and one or more consumers read and act on those messages asynchronously. This is useful for workloads that are slow, retriable, or that should not block an HTTP response (e.g., sending emails, processing uploads, triggering downstream pipelines).

How it's used here RabbitMQ is declared as a project dependency via the pika Python library. Pika is the standard AMQP client library for Python and is the mechanism through which the application connects to and communicates with RabbitMQ.

Note: The specific publish/consume logic (which queues are used, what messages are sent, and what consumes them) was not exposed in the provided source files. The presence of pika as a dependency confirms RabbitMQ integration exists within the project.
