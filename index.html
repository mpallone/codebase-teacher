What is this?

httpbin is a self-contained HTTP testing service. It exposes roughly 50 endpoints that mirror your request right back at you — headers, body, query params, origin IP — and simulate a wide range of HTTP behaviors on demand: redirects, authentication challenges (Basic, Digest, Bearer), compression, caching, streaming, cookies, and random status codes.

There's no database, no external dependencies, no persistent state. Point any HTTP client at it and it tells you exactly what it received.

Why does it exist?

Testing an HTTP client properly is surprisingly hard without a cooperative server. You need a target that behaves predictably, supports edge cases, and requires zero infrastructure to spin up. httpbin fills that gap.

Who reaches for it: library authors building HTTP clients, QA engineers validating retry/backoff logic, developers debugging auth flows, CI pipelines that need a live server to run against.

Concrete example: Priya is writing a Python HTTP client library and wants to verify it correctly handles gzip-compressed responses. She runs docker-compose up, sends GET /gzip, and immediately gets a compressed JSON body back. Her test asserts the decoded payload contains "gzip": true. No mocks, no stubs, no fake server to maintain — the test is reproducible anywhere Docker runs. She then hits GET /status/200:0.7,503:0.3 to test her retry logic against a server that fails 30% of the time. Done in an afternoon.

Here's how a typical request moves through the system:

flowchart LR
    A[Inbound Request] --> B["core.py\n(Flask route)"]
    B --> C["helpers.py\n(parse + assemble)"]
    C --> D["structures.py\n(CaseInsensitiveDict)"]
    B --> E["utils.py\n(weighted_choice)"]
    B --> F["filters.py\n(compress + X-Runtime)"]
    F --> G[Outbound Response]

core.py is the orchestrator. For most requests it calls helpers.py to extract what was sent and build a reply, optionally calls utils.weighted_choice for randomized behavior, then relies on filters.py decorators to compress the output and stamp an X-Runtime header before anything leaves the process. A Flask after-request hook adds permissive CORS headers to every response automatically.

One infrastructure detail worth knowing early: streaming endpoints like /drip and /stream use time.sleep inside Python generators. Gevent workers let those sleeps yield the thread without blocking other connections — that's why the Gunicorn command always includes -k gevent.

1. System Overview

httpbin is a stateless HTTP inspection and testing service. Think of it as a programmable mirror for HTTP: you send it a request, it reflects the request data back to you as structured JSON, or it simulates a specific protocol behavior (authentication challenges, redirects, compression, streaming, caching, etc.) that you want your client to handle correctly.

Its primary use cases are: - Client library testing — verify that an HTTP client correctly follows redirects, handles auth challenges, decompresses bodies, manages cookies, etc. - Manual inspection — quickly see what headers, parameters, or body your tool is actually sending - Negative testing — simulate slow responses, random failures, or specific error status codes

The system exposes approximately 50 HTTP endpoints, has no database, no external service dependencies at runtime, and carries almost no persistent state. It is designed to be trivial to deploy as a standalone container.

3. Component Descriptions

3.1 core.py — Flask Application and Routes

The system's backbone. This single file contains: - Flask app object creation and configuration - All ~50 route definitions (view functions) - The set_cors_headers after-request hook that injects permissive CORS headers on every response - Optional Bugsnag wiring - Flasgger (Swagger UI) initialization

Every inbound request is dispatched here first. core.py delegates parsing to helpers.py, randomization to utils.py, and post-processing to decorators from filters.py. The view functions themselves stay thin — their job is to orchestrate, not implement.

Example — the /get endpoint:

@app.route("/get")
@filters.x_runtime
def view_get():
    return jsonify(get_dict("url", "headers", "args", "origin"))

The route handler calls helpers.get_dict() to assemble the response payload, wraps it with Flask's jsonify(), and the @filters.x_runtime decorator adds timing information transparently.

3.2 helpers.py — Request Parsing and Response Construction

Two distinct responsibilities live here:

Request parsing — extracting and normalizing data from an inbound Flask request: - get_headers() — reads request.headers and wraps them in a CaseInsensitiveDict so downstream code can look up headers regardless of capitalization - get_dict() — assembles the standard reflection payload: origin (client IP), headers, args (query string), form, files, json (parsed body), and url

Response construction — building pre-shaped response objects for complex protocol behaviors: - status_code(code) — returns a Response object with the correct body and any required headers (e.g., a Location header for 3xx codes) for a given HTTP status - digest_challenge_response(...) — constructs a 401 response with a WWW-Authenticate: Digest header, generating a server nonce and encoding it into a Set-Cookie for replay prevention

3.3 filters.py — Response Post-Processing Decorators

filters.py provides transparent post-processing via Python decorators. It uses the decorator library (rather than functools.wraps) to preserve the original function's signature — important for Flask's routing and for Flasgger's OpenAPI spec generation.

Decorator What it does
@gzip Compresses response body with gzip if client sends Accept-Encoding: gzip
@deflate Compresses response body with zlib deflate
@brotli Compresses response body with Brotli (via C extension)
@x_runtime Measures wall-clock time around the handler and injects X-Runtime: <ms> into the response

The compression decorators inspect Accept-Encoding before compressing — if the client doesn't advertise support, the raw body is passed through unchanged.

3.4 structures.py — CaseInsensitiveDict

A minimal data structure that wraps a dict with case-insensitive key access. HTTP headers are case-insensitive by specification (e.g., Content-Type and content-type are equivalent), but Python dicts are not. This class normalizes all key lookups to lowercase, preventing subtle bugs when header names vary in capitalization across clients.

d = CaseInsensitiveDict({"Content-Type": "application/json"})
d["content-type"]  # → "application/json"
d["CONTENT-TYPE"]  # → "application/json"

3.5 utils.py — weighted_choice

A single utility function that implements weighted random selection over a list of (item, weight) pairs. It builds a cumulative probability distribution and draws a single random number to select one item.

This is used by the /status/<codes> endpoint to simulate probabilistic or flaky server behavior:

GET /status/200:0.7,500:0.3
# Returns 200 ~70% of the time, 500 ~30% of the time

3.6 __init__.py — Package Entry Point

Re-exports core.py's public API (app) as the package surface. This allows the application to be referenced as httpbin:app — which is the Gunicorn entry point in both the Dockerfile and Procfile.

4.4 Streaming Responses

Streaming endpoints (/stream/<n>, /drip, /stream-bytes/<n>, /range/<n>) use Python generators wrapped in Flask streaming Response objects. Werkzeug sets Transfer-Encoding: chunked automatically, and the Gevent worker releases the GIL during time.sleep() calls so other connections can proceed concurrently.

sequenceDiagram
    participant C as Client
    participant G as Gunicorn + Gevent worker
    participant Core as core.py

    C->>G: GET /drip?duration=5&numbytes=100&delay=1

    G->>Core: WSGI dispatch
    Core->>Core: parse duration, numbytes, delay params
    Core->>Core: create generator function
    Core-->>G: Flask streaming Response(generator, 200)

    loop Every chunk interval (duration/numbytes seconds)
        Core->>Core: generator yields next byte chunk
        G-->>C: Transfer-Encoding: chunked frame
        Note over G: Gevent greenlet sleeps;
other requests proceed
    end

    Core->>Core: generator exhausted
    G-->>C: final chunk + connection close

Key point about Gevent: Gevent replaces Python's standard blocking I/O with cooperative, non-blocking equivalents. When /drip calls time.sleep(interval), the Gevent greenlet yields control back to the event loop instead of blocking an OS thread. This allows one process to handle hundreds of concurrent slow-drip connections.

6. Infrastructure Dependencies

Flask (Web Framework)

Flask is a lightweight Python web framework. It handles: - Routing — mapping URL patterns to Python functions - Request context — making request.headers, request.args, etc. available globally within a request - Response lifecycle — after_request hooks, jsonify(), streaming Response objects

httpbin creates one Flask app object in core.py, registers all routes on it, and passes it to Gunicorn as the WSGI entrypoint.

Gunicorn (WSGI Server)

WSGI (Web Server Gateway Interface) is Python's standard interface between a web server and a web application. Gunicorn is the WSGI server that sits in front of Flask in production — it handles the raw TCP connections and translates them into WSGI calls.

httpbin uses Gunicorn with the gevent worker class (-k gevent):

gunicorn -b 0.0.0.0:80 httpbin:app -k gevent

Gevent (Async I/O)

Gevent is a coroutine-based concurrency library for Python. It uses "greenlets" — lightweight cooperative threads — and monkey-patches Python's standard I/O so that blocking calls (like time.sleep(), socket reads) yield control to other greenlets instead of blocking an OS thread.

This is critical for httpbin's streaming endpoints (/drip, /delay, /stream). Without Gevent, a single /drip?duration=30 request would tie up one OS thread for 30 seconds, dramatically limiting concurrency. With Gevent, that request's greenlet sleeps cooperatively while other requests proceed.

Flasgger (Swagger UI)

Flasgger reads YAML docstrings from Flask route functions and generates an OpenAPI specification automatically. It serves: - GET / — Interactive Swagger UI where you can explore and try all endpoints in a browser - GET /spec.json — Machine-readable OpenAPI JSON

Werkzeug (HTTP Utilities)

Werkzeug is Flask's underlying HTTP toolkit. httpbin uses it directly for parsing Authorization headers in the Digest Auth implementation.

Brotli (Compression)

A modern compression algorithm developed by Google, generally achieving better compression ratios than gzip. httpbin supports it via a C extension (brotli package). It is used on the /brotli endpoint when the client sends Accept-Encoding: br.

Bugsnag (Optional Error Tracking)

Bugsnag is a cloud-based error monitoring service. When the BUGSNAG_API_KEY environment variable is set, httpbin imports bugsnag and bugsnag.flask to automatically capture and report unhandled exceptions to the Bugsnag dashboard. If the key is absent, the import is skipped entirely — there is no runtime overhead.

if "BUGSNAG_API_KEY" in os.environ:
    import bugsnag
    import bugsnag.flask
    bugsnag.configure(api_key=os.environ["BUGSNAG_API_KEY"], ...)
    bugsnag.flask.handle_exceptions(app)

Docker / docker-compose (Containerization)

The service ships with a Dockerfile and a docker-compose.yml for local development and container-based deployment:

8. Key Design Decisions

8.1 Stateless by Default, State via Cookies When Necessary

The entire service is stateless — there is no database, no in-memory session store, no Redis. This is a deliberate design choice that makes the service trivially deployable and scalable.

The one exception is Digest Authentication, which requires the server to remember a nonce it issued. Rather than introducing a session store, httpbin encodes the nonce in a cookie on the client. The client sends the cookie back on subsequent requests, and the server reads it to verify the nonce — server-side storage is avoided entirely.

Trade-off: The client must support cookies for Digest Auth to work, which is reasonable since real browsers and most HTTP libraries do.

8.2 All Configuration via Environment Variables

There are no config files (no settings.py, no .env loading). Every configurable behavior is gated on an environment variable:

Variable Effect
DEBUG Enables Flask debug mode
BUGSNAG_API_KEY Activates Bugsnag error reporting
BUGSNAG_RELEASE_STAGE Bugsnag environment tag (default: production)
HTTPBIN_TRACKING Enables usage tracking

This follows the 12-Factor App principle and makes the service easy to configure in any container orchestration environment.

8.3 Decorators for Post-Processing (Separation of Concerns)

Rather than embedding compression logic inside each route handler, filters.py implements compression as decorators. A route that needs to return a gzip-compressed response simply applies @filters.gzip:

@app.route("/gzip")
@filters.x_runtime
@filters.gzip
def view_gzip_encoded_content():
    return jsonify(get_dict(..., gzip=True))

The route function doesn't know or care about compression. This separation makes individual routes easier to read and test, and makes it straightforward to add new post-processing steps (e.g., a new compression algorithm) without touching route code.

The decorator library (rather than functools.wraps) is used deliberately to ensure the wrapped function's signature is preserved — Flask uses the function name for route registration, and Flasgger uses the docstring for OpenAPI spec generation. Naive functools.wraps usage can interfere with both.

8.4 Gevent for Streaming Concurrency

The choice of gevent as the Gunicorn worker class (rather than standard sync workers or threads) is driven by the streaming endpoints. A synchronous worker model would block an entire OS thread for the duration of a /drip?duration=30 or /delay/10 request. Gevent's cooperative concurrency allows one process to serve hundreds of long-lived streaming connections simultaneously with minimal memory overhead.

8.5 No External Runtime Dependencies

Beyond the Python packages it imports, httpbin makes zero network calls at runtime. It does not contact: - Any database - Any cache service (Redis, Memcached) - Any external API - Any message queue

This makes the service completely self-contained and removes an entire category of failure modes, making it a reliable test target.

8.6 Permissive CORS by Default

The set_cors_headers after-request hook fires on every response and injects:

Access-Control-Allow-Origin: *
Access-Control-Allow-Credentials: true
Access-Control-Allow-Methods: GET, POST, PUT, DELETE, PATCH, OPTIONS
Access-Control-Allow-Headers: *

This is intentional — httpbin is a testing tool and CORS restrictions would make browser-based testing unnecessarily difficult. This would be inappropriate for a production service holding real user data, but is the right default for a developer tool.

Handler: redirect_n_times (httpbin/core.py)
Produces: text/html

Description

Issues a chain of 302 Found redirects that the client must follow n times before reaching a final destination. On each hop, the server issues a redirect to /redirect/<n-1>. When n reaches 1, the final redirect points to /get, which returns a standard 200 OK JSON reflection of that last request.

This endpoint is useful for:

  • Testing that HTTP client libraries correctly follow redirect chains and honour configurable redirect-follow limits.
  • Verifying that intermediate proxies and caches properly propagate redirects.
  • Triggering redirect-loop detection in clients (by setting n beyond the client's maximum follow count).

Safety clamp: The server validates and clamps n to a configured maximum to prevent unbounded redirect chains from being used to exhaust resources. If n exceeds this maximum, the server may return a 4xx error or silently clamp the value.

Path Parameters

Parameter Type Required Description

Query Parameters

None.

Response

Content-Type: text/html

Each intermediate response carries a Location header pointing to the next step in the chain.

2 GET /redirect/2 302 → /redirect/1
3 GET /redirect/1 302 → /get
4 GET /get 200 OK (JSON reflection)

Content-Type text/html Minimal HTML body (browser-facing redirect page).
Access-Control-Allow-Origin * CORS header added by set_cors_headers.

Summary: Returns the specified HTTP status code, or randomly selects one from a weighted list when multiple codes are provided.

Description

DELETE /cookies/delete

Tags: Cookies

Summary: Deletes one or more cookies by name (specified as query parameters) and redirects to GET /cookies to confirm removal.

Description

Accepts cookie names as query parameter keys (the values are ignored). For each named cookie, the server issues a Set-Cookie response header that sets the cookie to an empty string with Max-Age=0, which instructs the client to immediately expire and delete it. Note: This endpoint uses the HTTP DELETE method, but some HTTP clients and browser <form> elements only support GET and POST. In those environments, you may need to simulate the DELETE method via a custom header (e.g., X-HTTP-Method-Override) or use a test client that supports all HTTP verbs.

GET /basic-auth/\<user>/\<passwd>

Tag: Auth Handler: basic_auth (httpbin/core.py)

Description

Challenges the requesting client to authenticate via HTTP Basic Authentication (RFC 7617). The expected credentials are embedded directly in the URL path, making this endpoint useful for testing how HTTP clients handle 401 Unauthorized challenge-response flows, credential prompts, and credential encoding.

When a request arrives without an Authorization header (or with incorrect credentials), the server responds with 401 Unauthorized and a WWW-Authenticate: Basic realm="Fake Realm" header. A conforming HTTP client will then prompt the user (or automatically resend credentials from its credential store) and re-issue the request with an Authorization: Basic <base64(user:passwd)> header. On success the server returns 200 OK.

Note: Credentials in the URL path are transmitted in plain text over HTTP. Use HTTPS in any real scenario.

Path Parameters

Name Type Required Description
user string ✅ The expected username. The server validates the submitted username against this value.
passwd string ✅ The expected password. The server validates the submitted password against this value.

Request Headers

Header Required Description
Authorization conditional Must be Basic <base64(user:passwd)>. Absent on the first (challenge-triggering) request; must be present on the credential-carrying retry.

Response Format

Content-Type: application/json

Field Type Description
authenticated boolean true when the supplied credentials match the URL-bound user and passwd.
user string The username that was successfully authenticated.

Responses

Status Code Condition Body
200 OK Credentials match the path parameters. JSON object (see below).
401 Unauthorized No Authorization header present, or credentials do not match. Empty body; WWW-Authenticate: Basic realm="Fake Realm" header is set.

Content-Length: 0

HTTP/1.1 200 OK
Content-Type: application/json
X-Runtime: 0.001234

WWW-Authenticate: Basic realm="Fake Realm"

Related Data Flows

This endpoint does not participate in the HTTP Request Reflection flow. It implements its own standalone challenge-response cycle:

  1. Client sends unauthenticated GET.
  2. Server returns 401 with WWW-Authenticate: Basic.
  3. Client re-issues the request with a Authorization: Basic header.
  4. Server decodes and compares credentials against the URL path parameters.
  5. Match → 200 JSON; mismatch → another 401.

Tag: Auth Handler: hidden_basic_auth (httpbin/core.py)

Description

Functionally identical to GET /basic-auth/<user>/<passwd> with one critical behavioral difference: when authentication fails (wrong or missing credentials), the server responds with 404 Not Found instead of 401 Unauthorized, and does not emit a WWW-Authenticate challenge header.

GET /digest-auth/\<qop>/\<user>/\<passwd>

Tag: Auth Handler: digest_auth_md5 (httpbin/core.py)

Description

Implements the full HTTP Digest Authentication challenge-response protocol (RFC 7616) using MD5 as the fixed hash algorithm. This is the simplest variant of the three digest auth endpoints.

Digest auth is substantially more complex than Basic auth: rather than sending credentials in plaintext, the client computes an HMAC-style hash (HA1:HA2 hash chain) using a server-issued nonce, the requested URI, and the user's credentials. This prevents credential replay across different nonces and URIs.

First request (no Authorization header): The server generates a fresh nonce, writes it into a Set-Cookie header for replay-prevention tracking, and returns 401 Unauthorized with a WWW-Authenticate: Digest challenge header.

Second request (client retries with credentials): The server: 1. Reads the nonce from the request cookie and verifies it matches the Authorization header's nonce field. 2. Parses the Authorization header into its component credential fields. 3. Computes the expected MD5-based HA1:HA2 hash chain using the bound credentials and the received qop, nonce, nc, cnonce, and uri values. 4. Compares the computed hash to the client-supplied response field.

Path Parameters

Name Type Required Description
qop string ✅ Quality of Protection. Must be auth or auth-int. Controls whether the request body is included in the hash computation. auth-int provides integrity protection over the message body.
user string ✅ The expected username.
passwd string ✅ The expected password.

Algorithm: Always MD5 for this endpoint variant. Use /digest-auth/<qop>/<user>/<passwd>/<algorithm> to specify SHA-256 or SHA-512.

Request Headers

Header Required Description
Authorization conditional Digest scheme credential string produced by the client after receiving the challenge. Contains: username, realm, nonce, uri, nc, cnonce, response, qop.

Response Format

Content-Type: application/json (on success)

Field Type Description
authenticated boolean true on successful credential verification.
user string The authenticated username.

Responses

Status Code Condition Headers
200 OK Hash comparison succeeds. —
401 Unauthorized Initial challenge (no Authorization), stale nonce, or credential mismatch. WWW-Authenticate: Digest realm=..., nonce=..., qop=..., algorithm=MD5 and Set-Cookie: nonce=<value>.

    nonce="abc123deadbeef",
    qop="auth",
    algorithm=MD5
Directive Description
realm Authentication realm string (displayed to users).

Field Description

nonce Must match the nonce from the challenge (also validated via cookie).

  1. View checks for Authorization header.
  2. If absent → helpers.digest_challenge_response() issues 401 with WWW-Authenticate: Digest + Set-Cookie: nonce.

Tag: Auth Handler: digest_auth (httpbin/core.py)

Description

The most complete and configurable variant of the Digest authentication endpoints. Adds stale nonce simulation on top of the algorithm-aware behavior from the previous endpoint. The stale_after parameter controls how many successful authentications a given nonce is valid for before it is marked stale and a fresh challenge is issued with stale=true.

This endpoint is essential for testing how HTTP clients and libraries handle the RFC 7616 stale-nonce flow, where a server signals that the client should silently re-authenticate with a new nonce rather than treating the 401 as a credential failure.

The Stale Nonce Flow

When stale_after is set to an integer N:

  1. Requests 1 through N are accepted normally (each increments the nonce_count cookie).
  2. On request N+1, the server issues a new 401 challenge with stale=true in the WWW-Authenticate header.
  3. A conforming client must silently retry with fresh credentials against the new nonce without prompting the user.
  4. An stale=true response carries the same semantics as "your nonce expired, please re-authenticate" rather than "your password is wrong."

Setting stale_after to never (the default) disables stale behavior entirely — nonces never expire.

passwd string ✅ — The expected password.
algorithm string ✅ MD5 Hash algorithm. MD5, SHA-256, or SHA-512.
stale_after string ✅ never Integer string specifying the number of successful authentications before the nonce is invalidated and stale=true is issued. Use never to never expire.

The stale=true Directive

When stale=true appears in WWW-Authenticate, RFC 7616 requires the client to: - NOT prompt the user for new credentials (the credentials themselves are still valid). - Silently re-issue the request using the new nonce from the same challenge response.

Description

Streams a configurable number of bytes to the client at a controlled rate over a specified duration, optionally preceded by an initial delay. Unlike /delay/<n> which blocks and then responds all at once, /drip continuously feeds data to the client in small chunks across the lifetime of a long-lived connection.

Use this endpoint to test: - Streaming response consumption — verify your client can handle chunked/streaming HTTP bodies. - Progressive download logic — parse data as it arrives rather than buffering the full body. - Keep-alive connection behavior under slow producers. - Client-side streaming timeouts — distinguish between "connection timeout" and "read timeout." - Proxy/load balancer behavior for long-lived streaming connections.

GET /cache

Tag: Response inspection Handler: cache (httpbin/core.py)

Description

Simulates HTTP conditional caching behavior as defined in RFC 7232. The endpoint inspects the request for If-Modified-Since and If-None-Match conditional headers:

  • If either conditional header is present (regardless of the actual value): the server treats the request as a cache validation request and responds with 304 Not Modified with an empty body, simulating a "cache hit."
  • If neither conditional header is present: the server assembles and returns a full 200 OK JSON response with Last-Modified and ETag headers set, allowing the client to cache the response and use those headers in subsequent conditional requests.

This endpoint is part of the HTTP Caching Simulation data flow and enables engineers to test: - How HTTP clients and caches store and re-use ETag and Last-Modified values. - Whether clients correctly send If-None-Match / If-Modified-Since headers on repeated requests. - How proxies and CDNs handle 304 responses in a passthrough scenario. - Client cache invalidation logic.

Simplification: The server does not actually compare the header values to a real resource timestamp or ETag. The mere presence of either conditional header is sufficient to trigger a 304. This is intentional — it makes the behavior deterministic and fully controllable by the client.

Request Headers

Header Required Description
If-Modified-Since ❌ Standard HTTP conditional request header (RFC 7232 §3.3). If present (any value), the server returns 304.
If-None-Match ❌ Standard HTTP conditional request header (RFC 7232 §3.2). If present (any value), the server returns 304.

Content-Type: application/json (200 only; 304 has no body)

On 200 OK (cold request, no conditional headers):

headers object Normalized request headers as key-value pairs.
args object Query string parameters.
url string Full request URL.

Response headers on 200:

Header Description

ETag An opaque tag derived from request metadata. The client should store this and send it as If-None-Match on the next request.

On 304 Not Modified: Empty body. No Content-Type. No JSON.

Decision Logic

if (If-Modified-Since present) OR (If-None-Match present):

else:

    return 200 OK with JSON body + Last-Modified + ETag headers

How Flasgger Builds the Spec: Step-by-Step

Understanding this flow is important if you are adding new endpoints or modifying existing ones:

  1. Route registration — Flask registers view functions via @app.route(...) decorators in httpbin/core.py.
  2. Docstring parsing — Flasgger scans each view function's __doc__ string. The section after the --- delimiter is parsed as YAML. Fields like tags, produces, parameters, and responses populate the corresponding OpenAPI operation object.
  3. Path template translation — Flask path parameters (e.g., <int:n>) are translated to Swagger path parameters (e.g., {n} with type: integer).
  4. Spec assembly — All operations are merged into the global paths object. The info, host, basePath, schemes, and tags sections are populated from the swagger_config dict and Flasgger defaults.
  5. Serialisation — The assembled Python dict is serialised to JSON via Flask's jsonify() and cached.

Implication for contributors: If you add a new route without a properly formatted docstring YAML block, the route will still appear in paths but will have an empty responses and parameters object, making it undiscoverable via the Swagger UI. Always include the --- YAML section.

The GET /spec.json endpoint is outside all httpbin application data flows. It is served entirely by Flasgger. Specifically:

  • It does not invoke helpers.get_dict() or any reflection logic.
  • It is not decorated with filters.x_runtime.
  • It is not affected by the httpbin set_cors_headers after-request hook.
  • The spec is read-only — no POST/PUT/PATCH endpoint exists to mutate it at runtime. Changes require modifying view docstrings and restarting the server.

Gunicorn · compute

What It Is

Gunicorn (short for "Green Unicorn") is a production-grade WSGI HTTP server for Python applications on Unix. Unlike Flask's built-in development server, Gunicorn is designed to handle real traffic: it manages multiple worker processes, supports pluggable concurrency models, and handles process lifecycle management.

WSGI (Web Server Gateway Interface) is the standard Python interface between web servers and web applications. Gunicorn speaks WSGI to receive requests and forwards them to Flask.

How It's Used Here

Gunicorn is the process that actually listens for HTTP traffic and hands requests to Flask. It is configured in two places:

  • Dockerfile CMD — Gunicorn starts automatically when the Docker container launches.
  • Procfile — Declares the web process type for Heroku-style platform deployments.

Both configurations use the same command, ensuring consistent behaviour across deployment targets. The gevent worker class (see Gevent) is specified in both.

Gevent · compute

What It Is

Gevent is a coroutine-based Python networking library that enables high-concurrency I/O without the complexity of explicit async/await programming. It uses greenlets — lightweight cooperative threads — and monkey-patches Python's standard blocking I/O so that many operations can run concurrently within a single OS thread.

When Gevent is used as a Gunicorn worker class, each worker can handle many simultaneous connections efficiently, making it well-suited for applications that spend time waiting (on I/O, sleep calls, or streaming data).

How It's Used Here

Gevent is specified as the Gunicorn worker class in both the Dockerfile CMD and the Procfile. This is a deliberate choice driven by httpbin's endpoint behaviour:

  • /stream, /stream-bytes — Stream large responses incrementally.
  • /drip — Slowly drips data to the client over time.
  • /delay — Sleeps for a specified number of seconds before responding.
  • /range — Streams a range of bytes.

Without Gevent, a time.sleep call in one request would block an entire Gunicorn worker from handling any other requests. With Gevent, the sleeping greenlet yields control and the worker serves other connections in the meantime.

Tox · compute

What It Is

Tox is a Python testing automation tool. It reads a tox.ini configuration file and, for each defined test environment, creates a fresh isolated virtual environment with the correct Python version, installs the specified dependencies, and runs the configured test commands. This makes it easy to verify that an application works correctly across multiple Python versions without manually managing virtual environments.

How It's Used Here

tox.ini defines four environments:

  • py27, py36, py37 — Test environments. Each installs the project and runs python test_httpbin.py against its respective Python interpreter.
  • release — A separate, manually-triggered environment used to build distribution packages (sdist and wheel) and upload them to PyPI using twine. This is the release pipeline.

Travis CI calls tox (selecting the environment via the TOXENV variable) to run tests in CI. The release environment is used by maintainers to publish new versions.

Bugsnag · monitoring

What It Is

Bugsnag is a cloud-based error monitoring and crash reporting service. It captures unhandled exceptions that occur in a running application, records diagnostic context (stack traces, environment details, release stage), and sends alerts to developers. This provides visibility into production errors that might not surface in logs alone.

How It's Used Here

Bugsnag is an optional integration — it activates only if the BUGSNAG_API_KEY environment variable is set at application startup. This means the application runs without any monitoring in development (no API key configured) and gains error reporting in production (API key injected via environment).

The integration is wired into Flask in httpbin/core.py: at startup, the application checks for BUGSNAG_API_KEY; if present, it imports bugsnag and bugsnag.flask and attaches Bugsnag's exception handling middleware to the Flask app. From that point on, any unhandled exception is automatically captured and reported.

werkzeug.exceptions.NotFound (standard HTTP 404s) is intentionally excluded to avoid noise from routine missing-resource requests.
